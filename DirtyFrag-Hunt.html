Must-know caveat: Dirty Frag is not a race condition. Unlike Dirty Pipe (CVE-2022-0847), there is no timing window for the exploit to lose — the PoC takes any local user from uid=1000 to uid=0 in a single command with near-100% reliability. Assume every Linux host you have, including container hosts and CI runners, is one unprivileged shell away from root until patched or compensating modules are blacklisted. Container workloads inherit host kernel exposure: a compromise of any container that can create AF_KEY, XFRM netlink, or AF_RXRPC sockets (the default for unconstrained Docker, containerd, and most Kubernetes pods) escalates to host root.

Kill Chain

  1. Foothold: Attacker already has unprivileged local shell on Linux host or inside a container that can create AF_KEY/XFRM/AF_RXRPC sockets (the default for most pods/containers)
  2. Module load (auto): Opening an AF_RXRPC socket auto-loads rxrpc.ko; setting up an XFRM SA auto-loads esp4.ko/esp6.ko
  3. Page reference: vmsplice(2) a page from the page-cache of a setuid binary (typically /usr/bin/su) into an attacker-owned pipe as a paged fragment
  4. SA setup: Configure an XFRM Security Association with attacker-chosen cipher, key, and SPI via AF_KEY or XFRM netlink (look for XFRMA_REPLAY_ESN_VAL)
  5. In-place decryption: splice(2) the pipe into a UDP socket with ESP encapsulation; send a crafted ESP datagram to loopback. The kernel decrypts in place — into the borrowed /usr/bin/su page
  6. Shellcode landed: 192-byte x86_64 stub (setuid(0); setgid(0); execve("/bin/sh")) now lives in the in-RAM page-cache copy of /usr/bin/su
  7. Root: Attacker runs /usr/bin/su — the patched (in-cache) binary executes as root and drops a root shell. On-disk su is untouched; reboot or page-cache eviction clears the corruption
  8. RxRPC alternative: When user-namespace restrictions block XFRM SA setup (Ubuntu default), the chain pivots to RxRPC: open socket(AF_RXRPC), register an rxkad key via add_key("rxrpc", ...), and use the same splice-into-socket primitive

4.CQL Hunt Queries

All queries target Falcon LogScale / Next-Gen SIEM with the Linux sensor reporting. Each query is filtered to event_platform = "Lin" where applicable. HUNT-1 and HUNT-2 are the highest-signal entry points — module load and AF_RXRPC socket creation are nearly free pivots into the kill chain. HUNT-4 (root shell from non-root parent) is the definitive yes/no.

Linux telemetry shape varies. CrowdStrike's Linux event names depend on sensor version and on whether you have the Linux Cloud Workload Protection module enabled. The queries use regex alternation (e.g. /KernelModuleLoaded|SuspiciousKernelModuleLoaded|LinuxKernelModuleLoaded/) to span the common variants. Confirm against your tenant by running * | groupBy(#event_simpleName) | sort(_count, order=desc) over a 1-day window on a Linux-rich host first — pick the matching event name and tighten the query.
HUNT-1 Anomalous load of esp4 / esp6 / rxrpc kernel modules
MITRE: T1068 (precursor) FP Risk: Low Confidence: High Timeframe: Last 14 days Requires: Falcon Linux sensor; kernel-module-load telemetry

Hypothesis: The PoC implicitly loads the vulnerable module the first time it calls socket(AF_KEY), setsockopt(IP_XFRM_POLICY), or socket(AF_RXRPC). Hosts that have never used IPsec or AFS should never load these modules; if you see one of them appear, treat it as a Dirty Frag precursor.

FP notes: IPsec endpoints (strongSwan, Libreswan, OpenVPN-IPsec) load esp4/esp6 at boot. AFS clients (OpenAFS, kafs) load rxrpc. Build an allowlist of those hosts and exclude. Everything else is a strong signal.

// HUNT-1: Dirty Frag precursor - esp4/esp6/rxrpc module load
// MITRE: T1068 Exploitation for Privilege Escalation

#event_simpleName = /KernelModuleLoaded|SuspiciousKernelModuleLoaded|LinuxKernelModuleLoaded/
| event_platform = "Lin"
| ImageFileName = /\/(esp4|esp6|rxrpc|xfrm_user|ipcomp4|ipcomp6)\.ko(\.xz|\.zst|\.gz)?$/i
   or ModuleName = /^(esp4|esp6|rxrpc|xfrm_user|ipcomp4|ipcomp6)$/i
| ts := @timestamp
| groupBy([ComputerName, UserName, ModuleName, ImageFileName],
          function=[count(), collect([ts, aid, ParentBaseFileName, CommandLine])])
| sort(_count, order=desc)
HUNT-2 AF_RXRPC socket created by non-AFS process
MITRE: T1068 FP Risk: Low Confidence: High Timeframe: Last 7 days Requires: Falcon Linux sensor with socket-syscall telemetry (NGSIEM / Falcon for Cloud workloads)

Hypothesis: AF_RXRPC (socket family 33) has no legitimate userspace consumer outside the AFS toolchain (afsd, kafs-client, fs, vos). Any other process opening that socket family is almost certainly running the Dirty Frag RxRPC path.

FP notes: Add the AFS daemons in your environment to the exclusion list. Most enterprises don't run AFS at all — if that's you, there are no FPs.

// HUNT-2: AF_RXRPC socket created by unexpected process
// MITRE: T1068 / Dirty Frag RxRPC variant

#event_simpleName = /SocketCreate|LinuxSocketCreate|ProcessSocketCreated/
| event_platform = "Lin"
| SocketFamily = "AF_RXRPC" or SocketDomain_decimal = 33 or SocketAddressFamily = 33
| ImageFileName != /\/(afsd|kafs-client|fs|vos|bos|pts|venus)$/i
| ts := @timestamp
| groupBy([ComputerName, UserName, ImageFileName, CommandLine, ParentBaseFileName],
          function=[count(), collect([ts, aid, TargetProcessId_decimal])])
| sort(_count, order=desc)
HUNT-3 vmsplice + splice + sendmsg burst from unprivileged PID
MITRE: T1068 FP Risk: Medium Confidence: Medium Timeframe: Last 7 days Requires: Falcon syscall telemetry or auditd integration; this hunt is heaviest on telemetry — consider running on a small canary first

Hypothesis: The exploit's load-bearing pattern is a single process performing vmsplice, splice, and sendmsg against an XFRM/UDP-ESP socket inside <1 second. Legitimate use of those three syscalls together by a non-root, non-system process is vanishingly rare.

FP notes: Container runtimes, copy-on-write filesystems, and some media servers use splice for zero-copy I/O. Restrict to non-system UIDs (UID ≥ 1000 on most distros) and exclude containerd/runc/dockerd parents.

// HUNT-3: vmsplice + splice burst from same unprivileged process
// MITRE: T1068 - Dirty Frag primitive setup
// NOTE: requires syscall-level telemetry. Adjust event name to match your
// sensor (Falcon Linux sensor emits SyscallEvent variants; auditd-forwarded
// data uses AuditdSyscall). The hunt looks for any single process emitting
// all three syscalls inside the search window.

#event_simpleName = /SyscallEvent|LinuxSyscall|AuditdSyscall/
| event_platform = "Lin"
| SyscallName = /^(vmsplice|splice|sendmsg)$/i
| UID_decimal >= 1000
| ImageFileName != /\/(containerd|runc|dockerd|kubelet|crio|systemd)/i
| CommandLine != /(ffmpeg|nginx|tar\s|cp\s|rsync)/i
| groupBy([aid, ComputerName, UserName, TargetProcessId_decimal, ImageFileName],
          function=[count(as=syscall_count),
                    collect(SyscallName, limit=50),
                    min(@timestamp, as=first_seen),
                    max(@timestamp, as=last_seen)])
| burst_window_ms := last_seen - first_seen
| syscall_count >= 3 and burst_window_ms < 60000
| sort(syscall_count, order=desc)
HUNT-4 Root shell from non-root parent via /usr/bin/su (the smoking gun)
MITRE: T1548 / T1068 FP Risk: Low Confidence: High Timeframe: Last 30 days Requires: Falcon process telemetry (Linux)

Hypothesis: The terminal step of the public Dirty Frag PoC is the victim user invoking /usr/bin/su. Because the in-cache binary was rewritten with shellcode, the resulting process is uid=0 without a corresponding PAM success audit event. Look for /bin/sh or other shells spawning under su with effective uid=0 and parent uid != 0.

FP notes: Real su invocations with successful auth produce uid=0 children too; the discriminator is the missing PAM success. If you don't pipe syslog/auditd into LogScale, you'll need to triage manually — but the hit volume on a healthy fleet should be small enough to handle.

// HUNT-4: su execution producing root shell from non-root parent
// MITRE: T1548 Abuse Elevation Control Mechanism / T1068

#event_simpleName = /ProcessRollup2|SyntheticProcessRollup2/
| event_platform = "Lin"
| ParentImageFileName = /\/usr\/bin\/su$/i
| EffectiveUID_decimal = 0 or UID_decimal = 0
| ImageFileName = /\/(bash|sh|dash|zsh|ksh)$/i
// Parent of su was non-root - this is the suspicious shape
| ParentUID_decimal != 0
| ts := @timestamp
| groupBy([ComputerName, UserName, ParentUID_decimal, ImageFileName, CommandLine],
          function=[count(), collect([ts, aid, TargetProcessId_decimal, ParentProcessId_decimal])])
| sort(_count, order=desc)
HUNT-5 XFRM netlink configuration from unprivileged UID
MITRE: T1068 (ESP variant) FP Risk: Low Confidence: High Timeframe: Last 7 days Requires: auditd or Falcon NGSIEM netlink visibility

Hypothesis: Configuring an XFRM Security Association via netlink (NETLINK_XFRM, family 6) from a process that isn't ip xfrm, charon (strongSwan), pluto (Libreswan), or another IPsec daemon is suspicious. The PoC uses XFRMA_REPLAY_ESN_VAL to install its attacker-controlled SA.

FP notes: IPsec management traffic from running IPsec daemons; allowlist by image path.

// HUNT-5: XFRM netlink configuration from non-IPsec process
// MITRE: T1068 / Dirty Frag ESP variant

#event_simpleName = /NetlinkSocketCreate|LinuxNetlink|SyscallEvent/
| event_platform = "Lin"
| (NetlinkFamily = "NETLINK_XFRM"
   or NetlinkProtocol_decimal = 6
   or (SocketProtocol_decimal = 6 and SocketDomain_decimal = 16))
| UID_decimal >= 1000
| ImageFileName != /\/(ip|charon|pluto|swanctl|starter|NetworkManager|systemd-networkd)$/i
| ts := @timestamp
| groupBy([ComputerName, UserName, ImageFileName, CommandLine, ParentBaseFileName],

| sort(_count, order=desc)
HUNT-6 Privilege transition shortly after splice/vmsplice activity
MITRE: T1068 / T1548 FP Risk: Low Confidence: High Timeframe: Last 14 days Requires: Falcon Linux process telemetry; auditd or syscall events for splice/vmsplice

Hypothesis: Correlate non-root splice/vmsplice activity on a host with the next privilege transition on the same host within 60 seconds. The exploit window is small and the temporal correlation is tight.

FP notes: Builds, package installers running as root after a non-root build step. Tighten by requiring the privilege transition to occur in the same process tree.

// HUNT-6: Privilege transition within ~60s of unprivileged splice activity
// MITRE: T1068 + T1548 - correlation hunt
// NOTE: selfJoinFilter binds both halves by aid so the same
// host's splice burst and uid=0 transition appear in the same group.
// Filter the result for groups that contain BOTH a splice/vmsplice event
// AND a uid=0 child where the parent was uid != 0.

#event_simpleName = /SyscallEvent|LinuxSyscall|ProcessRollup2|SyntheticProcessRollup2/
| event_platform = "Lin"
| selfJoinFilter(field=[aid], where=[
    {#event_simpleName = /SyscallEvent|LinuxSyscall/
     | SyscallName = /^(splice|vmsplice)$/i
     | UID_decimal >= 1000},
    {#event_simpleName = /ProcessRollup2|SyntheticProcessRollup2/
     | EffectiveUID_decimal = 0
     | ParentUID_decimal != 0
     | ImageFileName = /\/(bash|sh|dash|zsh|ksh)$/i}
  ])
| ts := @timestamp
| groupBy([aid, ComputerName, UserName],
          function=[count(), collect([ts, SyscallName, ImageFileName, CommandLine, UID_decimal, EffectiveUID_decimal])])
| sort(_count, order=desc)
HUNT-7 Post-exploitation: SSH key / cron / systemd unit dropped by newly-root process
MITRE: T1098.004 / T1053.003 / T1543.002 FP Risk: Medium Confidence: Medium Timeframe: Last 14 days Requires: Falcon file-write telemetry on Linux

Hypothesis: After landing root, attackers near-universally drop persistence: an SSH key, a cron entry, or a systemd unit. If the writing process's ultimate ancestor was non-root, that's the Dirty Frag follow-on.

FP notes: Configuration management (Ansible, Puppet, Salt) writes these paths constantly. Allowlist by parent process name and parent path.

// HUNT-7: Persistence artifact written by newly-elevated process
// MITRE: T1098.004 / T1053.003 / T1543.002

#event_simpleName = /FileWritten|NewFileWritten|LinuxFileWrite/
| event_platform = "Lin"

   or TargetFileName = /\/home\/[^\/]+\/\.ssh\/authorized_keys$/i

| UID_decimal = 0
| ImageFileName != /\/(ansible|puppet|salt-minion|chef-client|cfengine|cloud-init|systemd|apt|dpkg|rpm|yum|dnf|cloud-config)/i
| ts := @timestamp
| groupBy([ComputerName, UserName, ImageFileName, TargetFileName, CommandLine],
          function=[count(), collect([ts, aid, ParentBaseFileName])])
| sort(_count, order=desc)

7.Hardening Recommendations

Recommendations are tiered by deployment friction. Each control cites the MITRE mitigation (M-number) it satisfies and, where relevant, the CIS benchmark line item. The Immediate tier is the genuine emergency lever — everything else is hygiene that should have been there anyway.

Immediate (deploy today — before lunch if possible)

1. Blacklist the vulnerable modules on every Linux host that doesn't need them. This is the single highest-leverage control and is reversible. Apply to every Linux host that does not terminate IPsec (strongSwan, Libreswan, OpenVPN-IPsec) or use AFS.

MITRE: M1038 Execution Prevention. What it does: Refuses to load the vulnerable code into the kernel at all.

Run as root: 
sudo sh -c "cat > /etc/modprobe.d/dirtyfrag.conf <<'EOF'
# Dirty Frag (CVE-2026-43284, CVE-2026-43500) + Fragnesia (CVE-2026-46300) mitigation
install esp4 /bin/false
install esp6 /bin/false
install rxrpc /bin/false
install xfrm_user /bin/false
install ipcomp4 /bin/false
install ipcomp6 /bin/false
EOF
"
# Unload any modules already loaded (rmmod refuses if in use; that is fine)
sudo rmmod esp4 esp6 rxrpc xfrm_user ipcomp4 ipcomp6 2>/dev/null || true

# Verify
lsmod | grep -E 'esp4|esp6|rxrpc|xfrm_user|ipcomp' || echo "OK: modules not loaded"
Warning: This breaks IPsec and AFS on affected hosts. Audit first with lsmod | grep -E 'esp4|esp6|rxrpc' across the fleet to find the hosts that do need these modules; use the "No Impact to IPsec Method" below on those. Per AWS bulletin 2026-027 and Red Hat RHSB-2026-003, the extended module list (xfrm_user, ipcomp4, ipcomp6) closes the Fragnesia bypass.
2. On hosts that must keep IPsec working, disable unprivileged user namespaces instead. This blocks the ESP variant (CVE-2026-43284) without breaking IPsec.

MITRE: M1038 Execution Prevention / M1026 Privileged Account Management. What it does: Strips the kernel ability for non-root users to create XFRM SAs at all. 
echo "user.max_user_namespaces=0" | sudo tee /etc/sysctl.d/dirtyfrag.conf
sudo sysctl --system

# Verify
sysctl user.max_user_namespaces
Side effects: Breaks rootless containers (rootless Docker, rootless Podman), sandboxed browsers (Chrome/Firefox sandbox uses user namespaces on some distros), Flatpak/Snap sandboxing. Pilot on a small ring before fleet-wide. On Ubuntu 24.04+ a softer alternative is kernel.unprivileged_userns_clone=0.
3. Patch as soon as upstream lands. CVE-2026-43284 (esp4/esp6) was patched in mainline on 2026-05-08. CVE-2026-43500 (rxrpc) and CVE-2026-46300 (Fragnesia) are still rolling per distro. Track:
  • Red Hat: RHSB-2026-003 — expedited release in progress
  • Ubuntu: USN tracker for linux-image-generic
  • Amazon Linux: AWS Security Bulletin 2026-027
  • SUSE: SUSE-SU-2026-XXX (tracking)
MITRE: M1051 Update Software.

Short-term (1–2 weeks — deploy after pilot)

4. Tighten container/pod seccomp to block AF_KEY, XFRM netlink, AF_RXRPC. The default Docker seccomp profile already blocks AF_RXRPC but not AF_KEY or XFRM netlink configuration. Most Kubernetes clusters ship with seccomp set to Unconfined or RuntimeDefault which doesn't cover this.

MITRE: M1048 Application Isolation and Sandboxing. CIS: Docker Benchmark 5.21 (Default seccomp profile), Kubernetes Benchmark 5.7.2 (Seccomp profile set).

Add to your custom seccomp profile (JSON, syscall denylist), or for Kubernetes use the seccompProfile: type: Localhost with a profile that denies these: 
{
  "defaultAction": "SCMP_ACT_ALLOW",
  "syscalls": [{
    "names": ["socket"],
    "action": "SCMP_ACT_ERRNO",
    "args": [{
      "index": 0, "value": 18, "op": "SCMP_CMP_EQ"
    }]
  }, {
    "names": ["socket"],
    "action": "SCMP_ACT_ERRNO",
    "args": [{
      "index": 0, "value": 33, "op": "SCMP_CMP_EQ"
    }]
  }]
}
(Where 18 is AF_KEY / PF_KEY and 33 is AF_RXRPC.) Apply at pod-spec or runtime-class level.
5. Enable Linux Kernel Lockdown if your distro supports it. Lockdown=integrity blocks several primitives used by kernel exploits to chain to root; it doesn't directly close Dirty Frag, but it raises the cost of post-exploitation persistence (no /dev/mem writes, no module insertion from non-signed sources). RHEL 9+, Ubuntu 22.04+ (Secure Boot required).

MITRE: M1038 Execution Prevention. Verify with cat /sys/kernel/security/lockdown → should show [integrity] or [confidentiality].
6. Deploy a Falco rule (or auditd rule) for AF_RXRPC socket creation. The Sysdig TRT-published Falco rule (see §10 References) catches the RxRPC variant at its earliest signature.

MITRE: M1056 Pre-compromise (Detection-oriented mitigation). 
- rule: AF_RXRPC Socket Created by Unexpected Process
  desc: Detects creation of an AF_RXRPC (domain 33) socket by a non-AFS process
  condition: >
    evt.type=socket and evt.dir=< and evt.rawres >= 0 and
    (evt.rawarg.domain=33 or evt.arg.domain contains AF_RXRPC) and
    not proc.name in (afsd, kafs-client, fs, vos, bos, pts, venus)
  output: AF_RXRPC socket created by unexpected process (proc=%proc.name uid=%user.uid cmdline=%proc.cmdline)
  priority: CRITICAL
  tags: [linux, mitre_privilege_escalation, T1068, dirtyfrag]

Long-term / structural (1–3 months)

7. Live-patching coverage for production Linux fleets. If you don't already have it: KernelCare (TuxCare), Canonical Livepatch, Red Hat kpatch, or Oracle Ksplice. Live patching lets you close Dirty Frag without rebooting the fleet. For a 9-year window of vulnerable kernels with a deterministic PoC in the wild, reboot-required patching is a liability.

MITRE: M1051 Update Software. Bias to live patching for any tier that can't tolerate reboot windows (databases, K8s control planes, AI/ML training clusters).
8. Audit and reduce setuid binary surface. The PoC targets /usr/bin/su because every host has it; but any setuid-root binary works as the trigger. Inventory: 
sudo find / -xdev -perm -4000 -type f 2>/dev/null
Remove setuid from binaries that don't need it (e.g., chsh, chfn, newgrp on hosts where no one uses these features). Pair with file-integrity monitoring (AIDE, Tripwire, OSSEC) so any future on-disk modification of setuid binaries fires — even though Dirty Frag itself leaves the on-disk file alone, the next variant might not.

MITRE: M1022 Restrict File and Directory Permissions / M1042 Disable or Remove Feature or Program. CIS: CIS Distribution-Independent Linux 6.1.13 (Audit SUID executables).
9. Enforce non-root containers and read-only root filesystems by default. A container that runs as uid=0 and can write to its own filesystem amplifies the blast radius of any container escape, including this one. Set in your Kubernetes PSA/Pod Security Standards baseline: 
securityContext:
  runAsNonRoot: true
  runAsUser: 10001
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"]
MITRE: M1026 Privileged Account Management / M1048 Application Isolation. CIS: Kubernetes Benchmark 5.2.5, 5.2.6.
10. Add the "Dirty Frag-class" pattern to your detection backlog as a recurring class. Sysdig TRT's framing is right: Dirty Pipe, Copy Fail, Dirty Frag, and Fragnesia are all "in-place kernel processing over attacker-controlled paged buffers" — this is a recurring exploit class, not a one-off. Schedule HUNT-3 and HUNT-6 to run weekly so the next variant doesn't get a free pass.

Priority Gaps

  • T1070.006 Timestomp (effective): Dirty Frag's killer property — the on-disk /usr/bin/su is unchanged, so file-integrity tools (AIDE, Tripwire, IMA) won't flag it. Mitigate via in-memory detection: at periodic intervals, hash the in-cache copy of /usr/bin/su by reading /proc/self/exe from a guard process invoked as root, and compare against the packaged hash. Falcon doesn't do this natively; consider a custom Ninja-style script run hourly.
  • T1014 Rootkit (in-memory): Same root cause. Lockdown=integrity helps adjacent persistence techniques but not the in-memory corruption itself. Best mitigation is to never get exploited — emphasize the module-blacklist control.
  • T1082 / T1518 Discovery: Pre-exploit reconnaissance (reading /proc/version, lsmod) is impossible to distinguish from normal admin activity at scale. Accept as a gap; the rest of the chain catches the exploit well before exfil.
