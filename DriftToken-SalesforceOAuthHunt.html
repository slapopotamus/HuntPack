1. Executive Summary

Between August 8 and August 18, 2025, the threat actor tracked by Google Threat Intelligence Group (GTIG) as UNC6395 (also referenced as GRUB1; extortion claims later branded ShinyHunters) systematically exfiltrated CRM data from ~700 Salesforce customer tenants by abusing OAuth access and refresh tokens stolen from Salesloft's Drift third-party integration. There is no Salesforce platform vulnerability — the campaign is a textbook SaaS supply-chain breach: compromise one trusted integration, inherit its trust to every customer.

Public victims include Cloudflare, Zscaler, Palo Alto Networks, Proofpoint, Tenable, Cisco, Tanium, BeyondTrust, PagerDuty, SpyCloud, Google (limited), Avalara, Toast, and many more. The actor's primary intent was credential harvesting — Salesforce Case and CaseComment bodies are notoriously full of secrets pasted by customers (AKIA AWS keys, Snowflake tokens, xoxb- Slack tokens, JWTs, VPN/SSO URLs, passwords). UNC6395 ran the harvested data through TruffleHog and validated keys, kicking off a second wave of breaches against the orgs whose secrets had leaked through their own support tickets.

Root cause on the Salesloft side: a monthslong GitHub account compromise (Mar–Jun 2025) let the actor exfiltrate source code, pivot to Salesloft's AWS environment, and lift OAuth tokens from the Drift platform. On 2025-08-20 Salesloft and Salesforce revoked all Drift access/refresh tokens; on 2025-08-28 Drift was removed from the AppExchange entirely. The FBI IC3 FLASH dated 2025-09-12 chains UNC6395 (OAuth theft) with the companion UNC6040 activity cluster (vishing-driven malicious Connected-App authorization) — same defensive surface, both worth hunting.

Attack chain (vendor-side foothold → cross-tenant exfil)

  1. T−∞ (vendor side): Salesloft maintains a Drift platform that holds OAuth access + refresh tokens for every customer's Salesforce / Google Workspace / Slack / etc. integration.
  2. T−5mo: Salesloft GitHub org compromised. Source repos exfiltrated; GitHub Actions tampered.
  3. T−2mo: Actor pivots from GitHub-leaked secrets into Salesloft AWS; reads OAuth tokens from Secrets Manager / SSM Parameter Store.
  4. T+0 (customer side, Aug 8): Actor uses stolen refresh-token → exchanges for fresh access-token against Salesforce IdP → reads tenant via REST/Bulk API. No MFA challenge, no password prompt, no anomalous geo on the IdP login event — because the IdP login event is bypassed entirely by the refresh-token grant.
  5. T+1m: Recon SOQL — SELECT COUNT() FROM Account|Opportunity|User|Case against each tenant to size the prize.
  6. T+2m: Bulk export — User, Account, Opportunity, Case (the high-secret-density target).
  7. T+10m – T+1d: Offline TruffleHog / regex grep across exfiltrated text for AKIA, Snowflake URLs, xox[bpars]-, JWTs, VPN/SSO URLs, plaintext passwords.
  8. T+1d+: Actor deletes Bulk API AsyncApexJob rows to obscure activity (ineffective — EventLogFile persists).
  9. T+1d → T+30d: Validated secondary credentials used against the customer's AWS / Snowflake / Slack tenants. This is where most of the real damage happens.
  10. T+30d+: ShinyHunters extortion outreach to selected high-value victims.

Hunt-observable indicators

  • Tier 2: DELETE against AsyncApexJob by an integration user (anti-forensics).
  • Tier 3 (tripwire): Any read of the honey-Case record described in §10.
  • Tier 3 (post-breach): Endpoint command-line containing trufflehog against support-case exports — if the actor is dwelling internally, the secret-grep step happens somewhere.

6. CQL Hunt Queries (15)

Queries are written against the conventional CrowdStrike NG-SIEM Salesforce ingest (#repo = "salesforce") plus IdP ingests (Entra azure_signinlogs, Okta okta_system_log). Adjust the repo/field names to your environment. All queries default to a 90-day timeframe centered on the August 2025 window; tighten or widen as needed for your retention.

Run order: Q1 → Q2 → Q4 are the highest-fidelity first-pass triage (known UAs, known IPs, bulk Case export). If any of those return a hit, expand to Q3/Q5/Q9 to scope. Q7/Q8/Q11/Q12 are the standing-detection set (run continuously, not just for this incident).
CQL audit (2026-05-19, post-publish): the 15 queries below were re-reviewed against the CrowdStrike CQL reference and corrected. Each query has an AUDIT comment block below its <pre> describing what changed. Highest-impact fixes:
  • Q5 — broken case statement (missing field := value form); 7 OR'd regexes collapsed into one alternation.
  • Q8 — full rewrite; the original eventStats(... series=) isn't a real CQL function. Replaced with groupBy + baseline-subquery join.
  • Q11 — full rewrite; !in(values=<array-field>) doesn't work in CQL. Replaced with collect(separator="|") → regex membership-test pattern.
  • Q13 — same eventStats issue as Q8; replaced with self-join to baseline subquery. stddev → stdDev casing.
  • Q15 — invalid (field1 OR field2) = /regex/ syntax replaced with coalesce() + in().
  • Q4, Q9, Q10 — EVENT_TYPE/ENTITY_NAME regex lists converted to in() for indexed lookup; bucket(... series=) (not a real arg) replaced with arithmetic time-bucket + groupBy.
  • All queries — uppercase OR → lowercase or; inline (?i) → trailing /i flag; coalesce() default-value removed (CQL takes a list only — use a follow-on if()); collect(field=X) → collect(X) positional form; trailing literal hyphens inside character classes moved to start (lint cleanup).
  • Live-test fix #1 (Q2, Q5, Q10, Q11): collect() does not accept as= (valid args: multival, fields, separator, limit). The collected array carries the input field's name. To rename, drop as= and follow with rename(field=X, as=Y). Confirmed against LogScale v1.237.
  • Live-test fix #2 (Q4, Q8, Q11, Q13): LogScale v1.237's default() rejects field= as a named arg (suggestions: only value, replaceEmpty). Replaced every default(value=X, field=Y) with if(Y != "", then=Y, else=X, as=Y) — works identically and the if() signature is stable across LogScale versions. The crowdstrike-logscale skill ref documents default() imprecisely and should be updated.
Field-name caveat: queries use the native Salesforce EventLogFile column names (EVENT_TYPE, USER_NAME, SOURCE_IP, USER_AGENT, ENTITY_NAME, ROWS_PROCESSED, CONNECTED_APP_NAME). The CrowdStrike NG-SIEM Salesforce data-connector parser preserves these names but your tenant may prefix them (e.g. salesforce.event_type). Run one query against a real sample before deploying the full set; remap field names once if needed.

8. Salesforce Transaction Security & Real-Time Event Monitoring Backstops

8.1 Block bulk anonymous reads from integration users (TSP/Apex)

Transaction Security Policy (TSP) on the ApiEvent stream — block or 2FA-challenge when an integration user pulls more than N records of Case/CaseComment in a single window.

// Salesforce Transaction Security Apex Condition
// Setup → Transaction Security Policies → New (Event: ApiEvent, Action: Block)
public class DriftBulkCaseBlock implements TxnSecurity.EventCondition {
    public Boolean evaluate(SObject e) {
        ApiEvent a = (ApiEvent) e;
        // Block any Bulk/REST query on Case-family objects with rows > 5000 from a non-allowlisted integration user.
        Set<String> allowlist = new Set<String>{
            'analytics_etl@yourorg.com',
            'mulesoft_bridge@yourorg.com'
        };
        Set<String> sensitiveObjects = new Set<String>{
            'Case','CaseComment','FeedItem','FeedComment','EmailMessage','CaseHistory'
        };
        Boolean isBulkOrApi = a.EventType == 'ApiEvent' || a.EventType == 'BulkApi';
        Boolean targetsSensitive = sensitiveObjects.contains(a.QueriedEntities);
        Boolean overThreshold = a.RowsProcessed != null && a.RowsProcessed > 5000;
        Boolean notAllowlisted = !allowlist.contains(a.Username);
        return isBulkOrApi && targetsSensitive && overThreshold && notAllowlisted;
    }
}

15. Containment Runbook (T+0 → T+60 min)

Trigger: any of Q1 / Q2 / Q4 / honey-Case / Q15 fires or external notification (vendor, FBI, Mandiant, GTIG, press) names you as a victim.

T+0 → T+5 min — Confirm & freeze

  1. Open the alert. Confirm IOC matches against §12 — UA, IP, connected-app name. Take screenshots / preserve the EventLogFile entries.
  2. Page SOC manager + Salesforce admin owner (RACI §18).
  3. If the hit is a Drift / Salesloft / Drift Email connected-app: revoke its OAuth refresh tokens immediately (Setup → Connected Apps → OAuth Usage → Revoke; or §9.2 SOQL DELETE OauthToken).
  4. If the hit is an unknown connected-app: disable it (Setup → Connected Apps → Edit Policies → "Admin approved users are pre-authorized" → remove all profiles) and revoke its tokens.
  5. Snapshot active sessions: SELECT Id, UsersId, LoginType, SourceIp, CreatedDate FROM AuthSession WHERE IsCurrent = true. Revoke any matching the IOC IPs.

T+5 → T+15 min — Scope

  1. Run §9.1 (OauthToken list) — note every still-valid token's LastUsedDate; freeze anything < 24h.
  2. Run §9.6 + §9.7 — pull RestApi, BulkApi, ApiEvent, LoginEvent EventLogFile rows for ±7 days around the alert. Grep locally for full IOC list.
  3. Run Q4 across the last 180 days — anything > 10k Case rows pulled by any integration user is now an investigation target.
  4. Run Q5 (or 9.4 SOSL) — secret-pattern hunt across Case / CaseComment / FeedItem. Triage AKIA hits first, then Slack / Snowflake / JWT.
  5. Identify the data classification of every record touched in the alert window — needed for §19 regulator-notification clock.

T+15 → T+30 min — Contain

  1. Apply Login IP Range restriction to all integration users (H-01) — even if only one is suspect, blanket-tighten while investigating.
  2. Force re-auth on every active human session (Setup → Session Management → End All Sessions).
  3. Open ITSM tickets for credential rotation against every secret class Q5 identified (AWS keys, Snowflake tokens, Slack tokens, GitHub PATs). Rotation owners run in parallel.
  4. Notify your SaaS vendor PSIRT contact if the integration owner is a third party.
  5. Disable the affected integration in your IGA — kill its outbound credential refresh from Vault / Secrets Manager.

T+30 → T+60 min — Communicate & evidence

  1. Notify Legal & Privacy — they will need to assess regulator notification clocks (see §19). In the US, state breach laws and GDPR-equivalent triggers depend on the data classes touched.
  2. Preserve evidence: export full EventLogFile coverage to immutable storage (S3 Object Lock / Azure immutable blob). Don't rely on Salesforce retention.
  3. Open the formal IR case (Jira / ServiceNow security incident). Link this runbook execution + all screenshots.
  4. Start the §17 hunt-summary ticket — that's the artifact leadership reads.
  5. If extortion email is received (ShinyHunters pattern): do not engage. Forward to law enforcement (FBI IC3) and Legal. Begin §19 comms template prep.

T+60 min → T+7 days — Eradicate & recover

  1. Reinstall Salesforce H-01 through H-10 controls if not already in place.
  2. Force re-issuance of every secret class with non-zero exposure from Q5.
  3. Re-baseline Drift-replacement / chatbot integration's behavior — review scope, IP allowlist, token TTL before re-enabling.
  4. Schedule the §11 validation plays against the sandbox to prove the new controls fire correctly.
  5. Submit detection-engineering tickets for any IOC missed in the first pass (the next variant will look similar but UA/IP-shifted — your detection set must be resilient).
