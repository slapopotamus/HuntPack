1.Executive Summary

IronWorm is a custom Rust implant distributed via a wave of 37 trojanized npm package versions published from a single compromised maintainer account. It is a credential-theft engine first and a worm second: it harvests everything from a developer's keychain, then weaponizes those credentials to push itself into source repositories and re-publish trojanized packages, infecting the next victim.

The infection executes the moment a developer or CI runner types npm install. The malicious package.json declares a preinstall script pointing at a native ELF binary hidden under a tools/ directory (tools/setup; a .github/scripts/precheck variant also exists). The binary runs before dependency resolution — no build step, no user interaction. It is a 976 KB Rust release build wrapped in a UPX stub with its magic bytes overwritten to defeat signature unpackers, with per-call-site string encryption to resist reversing.

Once running it (a) sweeps 86 environment variables and 20+ credential file paths — AWS/GCP/Azure, npm, GitHub, SSH, Docker, Vault, Kubernetes, and the full 2026 generation of AI provider keys (Anthropic, OpenAI, Gemini, Cohere, Mistral, Groq, Perplexity, xAI); (b) injects a JavaScript hook into the Exodus desktop wallet to steal the password and seed phrase, weakening Electron security flags and exfiltrating to a loopback listener on 127.0.0.1:8738; (c) loads an eBPF kernel rootkit to hide its processes, sockets, and resist ptrace; (d) bootstraps Tor and beacons to /api/agent for tasking (upload secrets, drop files, remote shell); and (e) self-replicates — in CI it abuses npm Trusted Publishing (OIDC token exchange) to obtain a package-scoped token with no stored credential, then republishes trojanized releases. A second, not-yet-observed-in-the-wild path replaces a GitHub Actions workflow to serialize ${{ toJSON(secrets) }} into an artifact for credential-free exfiltration.

Defender priority: The highest-fidelity, lowest-effort detection is the install-hook → native ELF execution chain on Linux developer and CI hosts (Q1). Pair it with loopback wallet-exfil (Q4) and Tor-bootstrap (Q3) detections. Because the worm's true blast radius is your secrets, treat any confirmed hit as a full credential-rotation event for the affected identity — cloud, AI, npm, GitHub, SSH, and Vault/K8s — not just an endpoint cleanup. The eBPF rootkit's strongest stealth fails under kernel lockdown, which is also your fastest hardening win.

8.CrowdStrike LogScale CQL Hunt Queries

8 queries ordered by fidelity. Targets are Linux/macOS process, network, and DNS telemetry. CONF/FP/COST ratings from the quality pass. Tune exclusions to your known native-build toolchain before promoting any to alerting.

Q1. npm/yarn install hook spawning a native binary from node_modules

CONF: high | FP: medium | COST: low | Requires: Linux/macOS process command line

Looks for: a package-manager process spawning a native binary located under node_modules, especially at tools/setup or scripts/precheck. Accomplishes: catches the IronWorm initial-execution chain (attack-chain steps 1–2) — the single highest-fidelity signal.

Common FP: legitimate native modules run binaries during install — node-gyp, esbuild, @swc/core, sharp, prebuild-install, cypress. The tools/setup/precheck path filter keeps this tight; widen with care and exclude your known toolchain by ImageFileName.
// HUNT: npm/yarn install hook spawning a native binary from node_modules
// MITRE: T1195.002, T1059
// CONF: high
// FP: medium
// COST: low
// REQUIRES: Linux/macOS process telemetry with command line
// FALSE POSITIVES: node-gyp, esbuild, @swc/core, sharp, prebuild-install, cypress run native binaries at install
// TUNING: exclude ImageFileName for vetted native toolchain (e.g. /node-gyp$/, /esbuild$/, /prebuild-install$/)
#event_simpleName=/^(ProcessRollup2|SyntheticProcessRollup2)$/
| event_platform=/^(Lin|Mac)$/
| ParentBaseFileName=/^(npm|npm-cli\.js|npx|yarn|pnpm|node|sh|bash|dash)$/i
| ImageFileName=/\/node_modules\/.*\/(tools\/setup|scripts\/precheck|setup|precheck)$/i
| ImageFileName!=/\/(node-gyp|esbuild|prebuild-install)$/i
| table([ComputerName, UserName, ParentBaseFileName, ImageFileName, CommandLine, SHA256HashData], limit=200)

Q2. ELF execution from inside node_modules

CONF: medium | FP: medium | COST: low | Requires: Linux process telemetry

Looks for: any process whose image lives under a node_modules tree — a native binary shipped inside a package. Accomplishes: broader net than Q1 for variants that use different hook names or paths; pairs well with first-seen / low-prevalence enrichment.

Common FP: prebuilt binaries bundled by legit packages (e.g. .bin shims, @esbuild/*, @next/swc). Triage by hash prevalence (SHA256HashData) and whether the binary was newly written.
// HUNT: native binary executing from within a node_modules tree
// MITRE: T1195.002, T1027.002
// CONF: medium
// FP: medium
// COST: low
// REQUIRES: Linux process telemetry
// FALSE POSITIVES: @esbuild/*, @next/swc, .bin shims, other prebuilt native modules
// TUNING: join to first-seen/low-prevalence hash; exclude vetted publisher hashes
#event_simpleName=/^(ProcessRollup2|SyntheticProcessRollup2)$/
| event_platform="Lin"
| ImageFileName=/\/node_modules\//i
| ImageFileName!=/\/node_modules\/(\.bin|@esbuild|@next\/swc|esbuild|@swc)\//i
| groupBy([SHA256HashData, ImageFileName, ComputerName, UserName], function=count(), limit=500)
| sort(_count, order=asc)

Q3. Tor bootstrap on a developer / CI / server host

CONF: medium-high | FP: low-medium | COST: low | Requires: process + DNS telemetry

Looks for: a Tor binary executing, a torrc being referenced, or a download of the Tor expert bundle. Accomplishes: catches the C2 bootstrap (step 6). Tor on a build server or corporate dev host is almost always anomalous.

Common FP: developers legitimately running Tor Browser on personal-use workstations. Scope to servers/CI fleet or correlate with Q1/Q2 hits to remove noise.

12.Hardening — Tiered & Deployable

Every control is framework-cited: MITRE mitigation (why), the concrete setting (what), and where it lives. Prioritize the few high-leverage controls below over a long marginal list.

Immediate (this week — low risk)

  • Disable npm lifecycle scripts in CI — npm config set ignore-scripts true (or npm ci --ignore-scripts); pnpm: enable-pre-post-scripts=false. Kills the preinstall execution path outright. MITRE M1042 (Disable or Remove Feature); CIS Software Supply Chain 4.x.
  • Block Tor egress + the fallback host — deny outbound to Tor directory authorities/known guard nodes and DNS for torproject.org and temp.sh at the proxy/firewall on server & CI ranges. MITRE M1037 (Filter Network Traffic), M1031 (Network Intrusion Prevention).
  • Least-privilege GITHUB_TOKEN — set permissions: read-all (or per-job minimal) as the org/repo default; require explicit elevation. Neutralizes the workflow-swap secret-exfil path. MITRE M1018 (User Account Management); GitHub hardening baseline.
  • Enforce 2FA + scrutinize Trusted Publishing on all npm publishing accounts; require provenance/attestations. MITRE M1032 (Multi-factor Authentication).

Near term (1–4 weeks — pilot first)

  • Enable Linux kernel lockdown (lockdown=confidentiality) and disable unprivileged BPF (kernel.unprivileged_bpf_disabled=1) on build/server fleet. Directly breaks the rootkit's process/socket hiding. MITRE M1038 (Execution Prevention), M1028 (OS Configuration).
  • Run CI builds as non-root, least-capability — drop CAP_BPF/CAP_SYS_ADMIN; default seccomp profile. Removes the privilege the rootkit and many sweeps assume. MITRE M1026 (Privileged Account Management), M1028.
  • Dependency cooldown / minimum release age — block install of package versions newer than N days (e.g. 3–7) so freshly trojanized bursts can't reach builders. MITRE M1016 (Vulnerability Scanning) / supply-chain policy; CIS SSC.
  • Move secrets out of files into a broker — short-lived, scoped tokens from a secret manager / OIDC instead of long-lived ~/.aws/credentials, .npmrc, AI keys on disk. MITRE M1041 (Encrypt Sensitive Information), M1027 (Password Policies).

Strategic (1–3 months — architectural)

  • Private registry / curation gateway (e.g. JFrog Curation, internal proxy) that blocks known-malicious versions and enforces an allowlist; isolate builders from the public internet except through it. MITRE M1016, M1037.
  • Ephemeral, network-segmented build runners with egress allowlists and no standing credentials; destroy after each job. Limits worm propagation and credential blast radius. MITRE M1030 (Network Segmentation), M1028.
  • Pin GitHub Actions by commit SHA + require environment protection/approvals on secret-bearing jobs; restrict artifact download scope. MITRE M1018, M1047 (Audit).
  • Application allowlisting / integrity monitoring for wallet & Electron apps — detect tampering of webSecurity/sandbox/contextIsolation/nodeIntegration and CSP. MITRE M1038, M1045 (Code Signing).

PB-2. Harden Linux kernel against the eBPF rootkit

# Disable unprivileged BPF (persist via sysctl)
echo 'kernel.unprivileged_bpf_disabled=1' | sudo tee /etc/sysctl.d/90-bpf.conf
sudo sysctl --system
sysctl kernel.unprivileged_bpf_disabled      # expect: = 1

# Enable kernel lockdown (confidentiality) via kernel cmdline (GRUB)
#   add to GRUB_CMDLINE_LINUX:  lockdown=confidentiality
sudo sed -i 's/\(GRUB_CMDLINE_LINUX="[^"]*\)"/\1 lockdown=confidentiality"/' /etc/default/grub
sudo update-grub    # (Debian/Ubuntu)  |  grub2-mkconfig -o /boot/grub2/grub.cfg (RHEL)
# Verify after reboot:
cat /sys/kernel/security/lockdown    # expect: [confidentiality] highlighted

# Spot-check for hidden processes/sockets once lockdown unmasks them
ps -ef --sort=start_time | tail -40
ss -tanp | grep -E '127\.0\.0\.1:8738' || echo "no 8738 listener"
