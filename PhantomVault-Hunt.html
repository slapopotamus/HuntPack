Threat Writeup Summary

Source: Elastic Security Labs — Phantom in the Vault: An AI-Generated RAT That Lives in Obsidian (REF6598)

URL: https://www.elastic.co/security-labs/phantom-in-the-vault

Campaign Overview

Elastic Security Labs tracked REF6598 — a financially-motivated campaign targeting cryptocurrency and financial-services personnel via LinkedIn / Telegram social engineering. Operators impersonate venture-capital firms and direct victims to log into an attacker-controlled Obsidian Sync vault that pre-loads the Shell Commands community plugin with a malicious payload. Trust in the Obsidian/plugin ecosystem becomes the execution vector. The Windows kill-chain delivers PhantomPull (in-memory PE loader) which fetches PhantomPulse, an AI-assisted RAT that resolves its C2 from Ethereum blockchain transactions. A parallel macOS path uses AppleScript via osascript and falls back to a public Telegram channel for domain rotation.

The Attack Chain (Step-by-Step)

  1. Pretexting: Threat actor contacts target on LinkedIn posing as a VC; conversation pivots to Telegram. The target is given credentials to an Obsidian Sync account “for managing the deal database.”
  2. Plugin trust abuse: Victim signs in, accepts the prompt to enable community plugin sync, and the attacker-controlled obsidian-shellcommands plugin lands inside the vault at ...\.obsidian\plugins\obsidian-shellcommands\data.json. Note: the user must opt-in to plugin sync — this is social engineering, not a vulnerability.
  3. Stage 1 (Windows): A Base64-encoded PowerShell command in the plugin invokes Invoke-WebRequest against http://195.3.222.251/script1.ps1, writes tt.ps1 into $env:TEMP, then re-launches PowerShell with -ExecutionPolicy Bypass -WindowStyle Hidden -File.
  4. Stage 2 — PhantomPull: script1.ps1 uses BitsTransfer to fetch syncobs.exe from the same staging IP and reports status to /stuk-phase with tag OBSIDIAN (status strings: GFILE FOUND, GLAUNCH SUCCESS, GSESSION CLOSED). The PE is a 64-bit loader containing an AES-256-CBC encrypted resource (RT_RCDATA id 101), unpacked in memory with djb2 API hashing (seed 0x4E67C6A7), timer-queue sandbox evasion, and a reflective PE map.
  5. Stage 3 — downloader: PhantomPull creates mutex hVNBUORXNiFLhYYh, decrypts strings with a 16-byte rotating XOR key, and fetches the next-stage payload from panel.fefea22134.net (Cloudflare-fronted). Reflective DLL load via DllRegisterServer.
  6. Stage 4 — PhantomPulse RAT: Full-feature backdoor with command hashes for inject, drop, screenshot, keylog, uninstall, elevate (COM-based UAC bypass, T1548.002), and downgrade. Telemetry endpoints under /v1/telemetry/.
  7. Blockchain C2 dead-drop: Instead of a static C2, the RAT polls Etherscan / Blockscout (Ethereum L1, Base L2, Optimism L2) for the latest transaction to wallet 0xc117688c530b660e15085bF3A2B664117d8672aA, hex-decodes the transaction input, and XOR-decrypts it (key = the wallet address) to obtain the active C2 URL. This is infrastructure-agnostic; operators rotate C2 by submitting a new transaction.
  8. macOS variant: A Base64-encoded AppleScript piped through osascript writes a LaunchAgent at ~/Library/LaunchAgents/com.vfrfeufhtjpwgray.plist (KeepAlive=true, RunAtLoad=true) and pulls the second stage from 0x666.info, with Telegram channel t.me/ax03bot as a fallback for domain rotation. Payload is piped directly to osascript — no disk write.

Why It Matters

  • Plugin trust as the entry-point. Obsidian Sync + community plugins are a legitimate ecosystem; the malicious vault carries the payload as benign-looking JSON config. No exploit — the user is socially engineered into pre-trusting the tenant.
  • Blockchain dead-drop C2. No traditional indicator can sinkhole this: the operator never needs DNS or hosting they control. Detection has to focus on the process talking to Etherscan/Blockscout, not the destination.
  • AI-generated tradecraft. Verbose [STEP X/Y] debug strings, redundant fake integrity checks, and self-documenting code suggest LLM-assisted authorship — lowering the barrier to entry and forecasting more variants from the same toolchain.
  • Cross-platform parity. A Windows kill-chain and a fully separate macOS chain ship together. Defenders who hunt only Windows miss half the campaign.
  • Sinkhole-able by design. Anyone with the wallet address and the XOR key (= the wallet address itself) can publish a transaction that hijacks every implant's C2. This is a defender-side opportunity and a likely actor revision point.

Hunt Brief

  • Threat: Phantom in the Vault / REF6598 — PhantomPull loader + PhantomPulse RAT High Severity
  • Summary: Social-engineering campaign that abuses the Obsidian obsidian-shellcommands community plugin to launch a Base64 PowerShell loader, fetch the syncobs.exe PhantomPull stage via BitsTransfer, and ultimately deploy the PhantomPulse RAT — which resolves its C2 from Ethereum blockchain transactions. macOS variant uses osascript + LaunchAgent persistence with Telegram fallback.
  • Affected: Windows + macOS workstations belonging to crypto / fintech / VC-adjacent staff. Any user of Obsidian Sync with plugin sync enabled is in scope.
  • Actor: Unattributed; financially motivated; AI-assisted tooling.
  • CISA KEV: No (TTP-based campaign, not a CVE)

Hunt 3 — Obsidian Spawning a Script Interpreter
MITRE: T1204.002, T1059 · FP Risk: Very Low · Confidence: High
// HUNT: Obsidian.exe (or macOS Obsidian) parenting a script interpreter via the Shell Commands plugin
// HYPOTHESIS: REF6598's payload is executed when the obsidian-shellcommands plugin fires powershell/cmd/bash/zsh/osascript from Obsidian
// FP RISK: Very Low - Obsidian is a markdown editor and has no built-in scripting; any script child is suspicious
// FP NOTES: Power users may legitimately install Shell Commands and run benign scripts. Validate against admin/dev OUs only.
// CONFIDENCE: High
// TIMEFRAME: Last 90 days

#event_simpleName = /ProcessRollup2|SyntheticProcessRollup2/
| ParentBaseFileName = /^Obsidian(\.exe)?$/i
| ImageFileName = /\\(powershell(_ise)?\.exe|pwsh\.exe|cmd\.exe|wscript\.exe|cscript\.exe|mshta\.exe|bash|sh|zsh|osascript)$/i
| groupBy([aid, ComputerName, UserName, ImageFileName], function=collect([@timestamp, CommandLine, ParentCommandLine]))
| sort(_count, order=desc)
Hunt 4 — Writes to obsidian-shellcommands Plugin Directory
MITRE: T1546, T1204.002 · FP Risk: Low · Confidence: Med-High
// HUNT: Creation or modification of obsidian-shellcommands\data.json (where REF6598 ships its PowerShell payload)
// HYPOTHESIS: A vault that syncs in the attacker's plugin will write data.json under \.obsidian\plugins\obsidian-shellcommands\

Hunt 5 — PowerShell + BitsTransfer / Invoke-WebRequest to Suspicious Endpoint
MITRE: T1105, T1059.001, T1197 · FP Risk: Medium · Confidence: Medium
// HUNT: PowerShell loader signatures - Start-BitsTransfer / Invoke-WebRequest dropping to $env:TEMP
// HYPOTHESIS: script1.ps1 in REF6598 uses Start-BitsTransfer to pull syncobs.exe; the inline command uses iwr to pull tt.ps1
// FP RISK: Medium - admin tooling, software installers, and some MDM scripts use BITS/iwr
// FP NOTES: Common false-positives: Chocolatey/winget, MSP RMM agents, Intune scripts. Exclude signed admin pipelines.
// CONFIDENCE: Medium
// TIMEFRAME: Last 30 days

#event_simpleName = /ProcessRollup2|SyntheticProcessRollup2/
| ImageFileName = /\\(powershell|pwsh)\.exe$/i
| CommandLine = /(Start-BitsTransfer|BitsTransfer|Invoke-WebRequest|iwr\s|Invoke-RestMethod|irm\s|DownloadFile|DownloadString)/i
| CommandLine = /(env:TEMP|\$env:TEMP|\\Temp\\|\\AppData\\Local\\Temp\\|\\Users\\Public\\)/i
| CommandLine = /(\.ps1|\.exe|\.dll|\.zip)/i
| table([@timestamp, aid, ComputerName, UserName, ParentBaseFileName, ImageFileName, CommandLine])
| sort(@timestamp, order=desc)

Hardening Recommendations

Immediate — Target Phantom in the Vault TTPs

1. Govern Obsidian Sync & Community Plugins

  • Inventory all Obsidian users. Decide policy: permitted with managed plugin allow-list, permitted but no community plugins, or blocked outside approved knowledge-worker OUs.
  • Block Obsidian community plugin sync via DNS / proxy if not needed: obsidian.md plugin endpoints.
  • Educate staff: anyone receiving an "Obsidian Sync account" from an external party is being phished. Add to security-awareness curriculum.
  • For permitted users, document which plugins are approved. Shell Commands should be denied outside very limited dev/admin use.

2. Constrain PowerShell

  • Deploy Constrained Language Mode via WDAC/AppLocker for non-admin users.
  • Enable Script Block Logging (Event 4104), Module Logging (4103), Transcript Logging: HKLM\SOFTWARE\Policies\Microsoft\Windows\PowerShell\ScriptBlockLogging\EnableScriptBlockLogging = 1.
  • Forward 4104 logs to LogScale/SIEM and alert on decoded script content matching Start-BitsTransfer or Invoke-WebRequest with non-corporate destinations.
  • Disable PowerShell v2 engine: Disable-WindowsOptionalFeature -Online -FeatureName MicrosoftWindowsPowerShellV2*.

3. Block / Monitor BITS for Non-IT Use

  • Restrict BITS to IT/admin contexts via WDAC; alert on bitsadmin.exe or Start-BitsTransfer launched from user-mode parents (Office, browsers, Obsidian, Slack, Teams).
  • Egress filter: block BITS-style traffic over HTTP (not HTTPS) at the perimeter.

4. Close COM-Based UAC Bypass Surface

  • Set HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System\ConsentPromptBehaviorAdmin = 2 (Always Prompt on the secure desktop).
  • Set EnableLUA = 1 and PromptOnSecureDesktop = 1.
  • Deploy UAC level "Always Notify" via GPO for privileged users.
  • Monitor child processes of auto-elevated binaries (Hunt 12).

5. macOS Hardening

  • MDM-managed macOS: deploy a config profile that monitors / restricts ~/Library/LaunchAgents writes by non-allowed origins.
  • Block osascript network access via Endpoint Security framework / Jamf custom policy where feasible.
  • Enable XProtect Remediator and Endpoint Security telemetry into your EDR; ensure macOS sensors are reporting at the same fidelity as Windows.
  • Educate macOS users: any tool that asks them to paste a curl command into Terminal is a red flag.

6. Block the C2 Infrastructure

  • Firewall / DNS block panel.fefea22134.net, fefea22134.net, 0x666.info, 195.3.222.251 at perimeter and split-DNS.
  • Add thoroughly-publisher-troy-clara.trycloudflare.com to detect list (likely retired but a strong forensic anchor).
  • Do NOT block Cloudflare proxy IPs or *.trycloudflare.com wholesale — high collateral damage. Subdomain-level only.

Strategic — Reduce Attack Surface Generally

  • WDAC + LSA Protection: Roll WDAC in audit→enforce mode. Enable RunAsPPL=1 so PhantomPulse's keylog / credential modules can't read LSASS.
  • Reflective PE detection: Leverage CrowdStrike's native unsigned-module-load / unbacked-memory detection rules; ensure they are not suppressed.
  • Block direct egress to crypto block-explorer APIs from endpoints outside dev/trading OUs. Force through a sanctioned proxy or block entirely.
  • Email + LinkedIn awareness: training module specifically on VC / fundraising lures and Telegram pivots; HR + finance + comms teams are primary targets.
  • RMM & collaboration tool governance: any third-party tool that syncs config or plugins (Obsidian, Notion, VS Code workspaces, Cursor) needs a policy on community-plugin trust.
  • Mac LaunchAgent audit: scheduled review of ~/Library/LaunchAgents and /Library/LaunchAgents; alert on any non-vendor plist.

Opportunistic: Sinkhole the Blockchain C2

Because the malware fetches the most recent transaction to the dead-drop wallet and XOR-decrypts the input data using the wallet address as key, a defender can publish a transaction containing an encrypted sinkhole URL — redirecting all implants to a controlled endpoint. This is a coordination opportunity with law enforcement / industry CERTs (e.g., crypto-ISAC). Do not attempt without legal sign-off; some jurisdictions treat this as unauthorized interaction with the malware.
