1. Executive Summary

SocGholish is a JavaScript-based initial access loader that has operated since at least 2017. It infects users by injecting malicious content into legitimate compromised websites, presenting a convincing fake browser or software update prompt. When the user downloads and executes the lure file — a .js file directly or inside a ZIP — Windows Script Host (wscript.exe) runs the payload, which silently beacons back to C2, performs in-memory reconnaissance, and optionally retrieves second-stage malware.

Mustard Tempest (GOLD PRELUDE) operates SocGholish and sells access to groups including Indrik Spider (Evil Corp). In approximately one in four 2025 infections, a second-stage payload was delivered — most commonly MintsLoader loading ASyncRAT or StealC, and in some cases a Python-based backdoor used as ransomware precursor leading to RansomHub.

The majority of infections do not progress past initial reconnaissance. The highest-value defensive angles are: catching wscript.exe spawning from a browser with an outbound connection, detecting domain trust enumeration via nltest.exe (ransomware-precursor signal), and blocking wscript.exe as the default handler for .js files via GPO.

Defender priority: Focus on browser → wscript.exe process chains, nltest.exe domain enumeration, and scheduled task creation under user context. The nltest signal in particular should be treated as a ransomware precursor and escalated immediately.

3. Hunt Brief & Attack Chain

The nltest.exe signal specifically indicates hands-on-keyboard escalation toward ransomware.

8. CrowdStrike LogScale CQL Hunt Queries

Validate #event_simpleName, ParentBaseFileName, ImageFileName, CommandLine, and TargetFileName field names against your Falcon tenant schema before promoting detections. Parent process fields require full process tree telemetry to be enabled.

Q1. Browser Spawning wscript.exe — Core SocGholish Execution Chain

CONF: high | FP: low-medium | COST: low | Requires: process telemetry with parent tracking

Looks for: A browser process (Chrome, Edge, Firefox, IE) spawning wscript.exe as a child — the core execution chain when a user clicks a SocGholish fake update lure. Accomplishes: Catches the infection at the earliest detectable endpoint moment, before C2 check-in completes.

Common FP: Enterprise software updaters that use wscript.exe may trigger this. Tune by excluding known updater parent paths (e.g., approved IT management tooling) using ParentImageFileName allowlist.
// HUNT: Browser Spawning wscript.exe - SocGholish Execution Chain
// MITRE: T1059.007, T1204.001, T1189
// CONF: high | FP: low-medium | COST: low
// REQUIRES: ProcessRollup2 with parent process tracking
// FALSE POSITIVES: Enterprise software updaters using wscript; web-based management consoles
// TUNING: Exclude ParentImageFileName paths for known approved updaters

#event_simpleName=/ProcessRollup2|SyntheticProcessRollup2/i
| ImageFileName=/\\wscript\.exe$/i
| ParentBaseFileName=/(chrome|msedge|firefox|iexplore|browser|chromium|opera|brave|vivaldi)\.exe/i
| table([@timestamp, ComputerName, UserName, ParentBaseFileName, ParentImageFileName, ImageFileName, CommandLine, SHA256HashData])

Q2. wscript.exe Outbound Network Connection — C2 Beacon

CONF: high | FP: medium | COST: low | Requires: process + network telemetry

Looks for: wscript.exe making outbound network connections to external IPs — the C2 beacon and recon exfiltration step. Accomplishes: Detects the data exfiltration phase and can surface the C2 destination for blocking.

Common FP: Legitimate enterprise scripts that call external APIs or internal proxy relay addresses. Tune by excluding known internal IP ranges and trusted update domains. Pivot on RemoteAddressIP4 values for threat intel enrichment.
// HUNT: wscript.exe Outbound Network Connection - SocGholish C2 Beacon
// MITRE: T1048.003, T1102
// CONF: high | FP: medium | COST: low
// REQUIRES: NetworkConnectIP4, ProcessRollup2
// FALSE POSITIVES: Legitimate enterprise scripts calling external APIs, proxy relay
// TUNING: Exclude known internal CIDR ranges and approved update/CDN destinations

#event_simpleName=NetworkConnectIP4
| ImageFileName=/\\wscript\.exe$/i
| RemoteAddressIP4!=/^(10\.|172\.(1[6-9]|2[0-9]|3[01])\.|192\.168\.|127\.)/
| table([@timestamp, ComputerName, UserName, ImageFileName, RemoteAddressIP4, RemotePort, Protocol])

12. Hardening — Tiered & Deployable

Immediate

  • Remap .js file association via GPO. Change the default handler for .js files from wscript.exe to notepad.exe. This single control prevents SocGholish, Gootloader, and most JS-based loaders from executing on double-click. Deploy via GPO: HKCR\.js → set default to txtfile, or create a GPO Software Restriction Policy targeting the .js extension. Red Canary blog "Open with Notepad" covers implementation details.
  • Block wscript.exe / cscript.exe via ASR or AppLocker for standard users. Attack Surface Reduction rule: Block execution of potentially obfuscated scripts (GUID 5BEB7EFE-FD9A-4556-801D-275E5FFC04CC). For AppLocker, deny wscript.exe and cscript.exe for non-admin user groups.
  • Enable Script Block Logging and Script Content Capture in Falcon sensor policy and Windows PowerShell settings. This provides in-memory visibility into JS payload content.

Near Term

  • Deploy web filtering / DNS-based blocking to catch known SocGholish TDS infrastructure. Use threat intelligence feeds for current domain blocklists; these rotate, so automation is required.
  • Restrict scheduled task creation for standard user accounts via GPO (Computer Configuration → Windows Settings → Security Settings → User Rights Assignment → Create scheduled tasks — deny for non-admin groups).
  • Enforce MFA and conditional access on all accounts. SocGholish infections that lead to ransomware typically involve credential theft (StealC) as a stepping stone. Limiting credential reuse limits blast radius.
  • Monitor and restrict nltest.exe outside of known AD management tooling. Consider blocking nltest.exe for non-admin accounts entirely via AppLocker/WDAC.

Strategic

  • Implement a browser isolation or RBI solution for high-risk users (finance, HR, executives). Drive-by compromise requires the browser to render attacker-controlled content.
  • Adopt application control (WDAC or AppLocker) in audit mode, progressing to enforcement, to restrict unsigned script execution across the fleet.
  • Continuously validate JS execution controls via Atomic Red Team T1059.007 in your CI/CD-style detection pipeline. SocGholish is a persistent top-10 threat; controls need routine testing.

13. Deployable Playbooks

SocGholish Initial Triage

Scope: SOC L1 / endpoint owner
1. Confirm the alert: verify parent process (browser), child process (wscript.exe), and external network connection.
2. Identify lure file: check CommandLine of wscript.exe for the .js filename. Note any homoglyph characters.
3. Check for rad*.tmp artifacts in %TEMP% on the affected host (FileCreateInfo events or live file system).
4. Check for system discovery commands spawned by wscript.exe: whoami, ipconfig, nltest, net.
5. If nltest.exe /domain_trusts is present — ESCALATE IMMEDIATELY. This is a ransomware precursor.
6. Check for scheduled tasks created in the past 2 hours on the affected host.
7. Check for second-stage processes: python.exe, msiexec, or unusual loaders from wscript.exe parent.
8. Isolate host if: nltest is present, second-stage payload observed, or domain credentials may be at risk.
9. Preserve: browser history, %TEMP% contents, scheduled task XML, wscript.exe process tree, network connections.
10. Reset the affected user's credentials and revoke active sessions if domain credentials were exposed.

nltest Escalation Playbook

Scope: SOC Lead / IR — escalate immediately on nltest detection
1. Treat any nltest.exe /domain_trusts or /all_trusts from a non-admin account as a confirmed ransomware precursor.
2. Immediately contain the host via Falcon network containment.
3. Identify the user account and check for lateral movement: other hosts with the same user's credentials active.
4. Pull the full process tree on the affected host for the past 72 hours.
5. Check for credential staging: NTDS.dit access, LSASS dumps, SAM/SYSTEM copies.
6. Check for RansomHub or similar ransomware staging indicators: shadow copy deletion, event log clearing.
7. Notify IR lead and initiate incident response if any of the above are present.
8. Reset all domain credentials associated with the affected user. Revoke Kerberos tickets (klist purge / krbtgt reset if krbtgt is suspected).
9. Block outbound connections from the affected host while investigation is active.
