1. Executive Summary

Tampered Chef, as defined by Red Canary, is an Electron/Node.js threat that presents as legitimate desktop software while processing hidden command content embedded in normal-looking application data. Red Canary observed RecipeLister in June 2025 and later Calendaromatic in September 2025. The durable behavior is not a single static hash: it is the combination of user-installed, signed fake productivity apps, JavaScript-capable desktop frameworks, covert API response content, browser manipulation, and connections to a small set of campaign domains.

This hunt pack keeps three confidence layers separate: Red Canary narrow scope for RecipeLister and Calendaromatic; closely related Calendaromatic reporting from GuidePoint and Kroll; and broader TamperedChef-style malvertising clusters from WithSecure, Field Effect, Dataprise, HivePro, and Unit 42. Use the first layer for high-confidence incident response. Use the broader layer for exposure review, software inventory, and signer-based hunting.

This pack labels those broader pivots as adjacent, not automatically confirmed Red Canary Tampered Chef.

       v

       |
       v

       |
       v
[Desktop framework executes bundled JavaScript: Electron, Node.js, or NeutralinoJS]
       |
       v
[App contacts legitimate-looking API / content endpoint]
       |
       v
[Hidden command content is encoded in zero-width Unicode or homoglyph-like characters]
       |
       v
[JavaScript routine decodes and executes command content]
       |
       +--> [Open Chrome to adversary-controlled advertising/search pages]
       +--> [Change default search or new-tab behavior]
       +--> [Create registry Run key or scheduled task]
       +--> [Stealer/RAT branch in broader AppSuite-style reporting]
       |
       v
[Credential/session exposure risk: browser data, cookies, search hijack, remote payloads]

6. Native Audit-Log Hunts

Endpoint-first, but not endpoint-only: Tampered Chef often starts with local execution, but credential and browser-session exposure means identity, password-manager, and web-gateway logs matter for scoping.

Proxy / DNS / web gateway checks

HuntSearch logicAction
Download and control domainsSearch DNS/proxy for the domains in section 3, especially from unmanaged browser sessions or shortly before unknown signed app execution.Block domains, collect referrer/download chain, identify all users/devices.
Sponsored-result download trailLook for search-engine ad click redirects to fake PDF/calendar/recipe/manual utilities.Feed domains into blocklist and user awareness; search endpoint downloads folder for matching installers.
Browser hijack trafficNew recurring browsing to unknown search/ad domains immediately after Calendaromatic/RecipeLister execution.Reset browser policy/profile, remove app, rotate credentials if credential access is suspected.

Browser sync scopeChrome/Edge sync enabled for compromised profile; new device enrollments or profile sync events.Reset sync, revoke refresh tokens, review synced extensions and passwords.

11. Hardening

Immediate

  • Block known malicious domains and exact hashes from section 3.
  • Quarantine hosts with Calendaromatic, RecipeLister, AppSuite PDF, or known sample hashes.
  • Disable browser password saving for managed Chrome/Edge where feasible.
  • Revoke browser sessions and rotate credentials for confirmed credential-store access.
  • Enable ad-blocking or DNS filtering for sponsored-result malware patterns.

Near term

  • Deploy WDAC/AppLocker rules for approved software locations and publishers.
  • Block unsigned or unapproved signed executables from Downloads, Temp, and user-profile app folders.
  • Require software installs from managed portals and remove local admin where possible.
  • Monitor per-user Run keys and scheduled tasks created by non-IT processes.

Strategic

  • Adopt application allow-listing for productivity tools.
  • Centralize endpoint, DNS, proxy, identity, and password-manager logs for post-install scoping.
  • Train users to avoid sponsored download results for utilities and PDF tools.
  • Build signer reputation review into software intake and EDR triage.

# TamperedChef endpoint triage - run in an elevated PowerShell session
$names = @(
  "RecipeLister","Calendaromatic","calendaromatic-win_x64","ImageLooker",
  "PDF Editor","AppSuite Print","S3-Forge","DocuFlex","CrystalPDF",
  "Easy2Convert","PDF-Ezy","RocketPDFPro","ManualReaderPro","GoCookMate",
  "JustAskJacky"
)
$paths = @(
  "$env:LOCALAPPDATA\Programs",
  "$env:APPDATA",
  "$env:USERPROFILE\Downloads",
  "$env:TEMP"
)

foreach ($path in $paths) {
  if (Test-Path $path) {
    Get-ChildItem -LiteralPath $path -Recurse -ErrorAction SilentlyContinue |
      Where-Object { $n=$_.Name; $names | Where-Object { $n -like "*$_*" } } |
      Select-Object FullName, Length, LastWriteTime
  }
}

13. Containment Runbook

1. Isolate endpoint if Q9, Q10, exact-hash, or confirmed persistence hits are present.
2. Preserve installer, extracted app directory, browser profile metadata, scheduled task XML, Run key data, and process/network timeline.
3. Remove persistence only after evidence capture.
4. Revoke browser sessions and rotate saved passwords, API tokens, VPN credentials, and password-manager secrets exposed to the user.
5. Check proxy/DNS for all domains in section 3 across 180 days.
6. Review identity logs for first-seen IPs, impossible travel, new devices, token refresh, or MFA changes after install time.
7. Reimage confirmed stealer/RAT hosts where browser credential files were accessed.
8. Push exact hashes/domains to Custom IOC workflows and staged signer blocks to application control.

15. Hunt Summary Ticket

Title: Tampered Chef / TamperedChef-style fake productivity app exposure review

Scope:
Run CrowdStrike LogScale hunts Q1-Q18 for Windows endpoints over 90 days, with 180 days for exact hash/domain retro-hunts.

Primary questions:
1. Did any endpoint execute RecipeLister, Calendaromatic, ImageLooker, AppSuite PDF, or adjacent fake productivity apps?
2. Did any signed unknown app from the TamperedChef signer watchlist run or write files under user profile paths?
3. Did any endpoint contact known domains or spawn browser processes to suspicious URLs?
4. Did any fake app process read browser credential stores or create persistence?

Success criteria:
- All exact hash/domain hits are triaged.
- Confirmed installs have persistence removed, sessions revoked, and credentials rotated.
- Signer/application-control plan is staged with tested exclusions.
- Proxy/DNS and identity logs are reviewed for post-install activity.
