1. Executive Summary

Amber Albatross is a Red Canary–named Windows activity cluster active since January 2024 and ranked #1 most prevalent threat in Red Canary's 2026 Threat Detection Report, with 11.1% of Red Canary customers affected across 2025. It is the standout example of a 2024–2026 PUP-to-stealer ecosystem: a user installs what looks like a third-party app store (PC App Store) or a free PDF utility (PDFast, "Let's Compress", various rotating brands), an InnoSetup wrapper drops an early-stage binary, that binary downloads later stages via Base64-encoded PowerShell, and a Pyarmor-obfuscated PyInstaller executable lands as the final payload.

The final payload performs textbook stealer reconnaissance — WMIC enumeration of hypervisor / manufacturer / Windows updates, AV and firewall fingerprinting, browser-profile reads for six families (Edge, Firefox, Chrome, Chromium, Brave, Avast Browser), software-uninstall key enumeration (added in 2025), and a specific read of HKLM\SOFTWARE\Policies\Google\Chrome\CloudManagementEnrollmentToken to identify managed-browser-policy enrollment. Red Canary has not yet observed how Amber Albatross uses the harvested data — but the reconnaissance breadth is consistent with credential and corporate-context theft.

The operation evades sandboxes by requiring undocumented command-line arguments — --safetorun --channel=<hex> — that VirusTotal and most sandboxes never pass. Binaries are code-signed with EV certificates from a rotating cast of shell companies (11 known thumbprints from GlobalSign GCC R45 EV CodeSigning CA 2020 and one from SSL.com), so SmartScreen and naïve signature trust treat them as legitimate.

Execution chain (PC App Store variant — most common)

  1. T+0: User downloads + executes PCAppStore.exe (signed, branded as a legitimate-looking app store).
  2. T+seconds: PC App Store fetches and runs a signed bundled installer (signer is one of the 11 known EV thumbprints).
  3. T+seconds: Installer (InnoSetup-based) drops stage 1 to %LOCALAPPDATA%\<random>\ or %PROGRAMDATA%\<random>\.
  4. T+seconds: Stage 1 spawns cmd.exe → powershell.exe -e <base64> that downloads stage 2 from a C2 domain.
  5. T+1–5 minutes: Stage 2 (C++ historically; some Go in 2025) drops the Pyarmor-protected PyInstaller binary.
  6. T+5+ minutes: PyInstaller binary is executed via cmd.exe /c <numeric_name> --safetorun --channel=<hex>. Process name is fully numeric (e.g. 12345678), no extension.
  7. T+5–10 minutes (reconnaissance):
    • WMIC: computersystem get manufacturer,model, qfe get, bios get, cpu get
    • WMIC: hypervisor presence (Win32_BaseBoard.Manufacturer, virtualization heuristics)
    • Registry: HKLM\SOFTWARE\Policies\Google\Chrome\CloudManagementEnrollmentToken
    • Registry walk: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\* (2025+)
    • WMI: SELECT * FROM AntiVirusProduct, FirewallProduct (in root\SecurityCenter2)
    • File reads: each detected browser's User Data / Profile folder
  8. T+10+ minutes: C2 callback over HTTPS to one of the rotating domains; usage of stolen data not publicly observed by Red Canary as of pack date.

Execution chain (PDFast variant)

  1. User searches "PDF compressor free" → clicks ad / SEO result → lands on look-alike domain.
  2. Page offers download of a signed installer; user runs.
  3. Installer drops upd.exe to %TEMP% or %LOCALAPPDATA%.
  4. upd.exe spawns Base64-encoded PowerShell that downloads + executes the Pyarmor-protected stage.
  5. Rest of the chain matches the PC App Store variant from step 6 onward.

Endpoint-Observable Indicators (the hunt focus)

  • Tier 1: Process with command line containing both --safetorun AND --channel=.
  • Tier 1: File written to disk where Authenticode signer thumbprint matches one of the 11 known Amber Albatross thumbprints.
  • Tier 1: Process whose image base name is fully numeric (e.g., 123456) with no extension, executed by cmd.exe or powershell.exe.
  • Tier 1: Read of HKLM\SOFTWARE\Policies\Google\Chrome\CloudManagementEnrollmentToken by any process not in the allowlist (Chrome itself, MDM agents, IT enrollment scripts).
  • Tier 2: WMIC reconnaissance sequence (manufacturer / model / qfe / bios) from a single PID within 60 seconds.
  • Tier 2: Base64-encoded PowerShell (-e, -enc, -encodedcommand) by a non-IT-account that establishes external network connection AND writes an executable.
  • Tier 2: Network connection or DNS resolution to one of the 10 known Amber Albatross C2 domains (or the 2 earlier-stage ones).
  • Tier 2: PUP installer execution: PCAppStore.exe, BitDriverUpdater.exe, WinRiserSetup.exe, MalwareCrusher.exe.
  • Tier 2: upd.exe written to %TEMP% or %LOCALAPPDATA% by a signed installer parent.
  • Tier 3: Bulk read of multiple browser User Data folders by a non-browser process within a 10-minute window.
  • Tier 3: Long sequence of reads of subkeys under HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\* by a non-package-manager process.

Q1 ⭐ Process with --safetorun AND --channel= args
FP RISK: VERY LOW CONF: VERY HIGH COST: 🟢 LIGHT
MITRE: T1622, T1068

Hypothesis: The Amber Albatross final-stage PyInstaller binary requires the undocumented argument pair --safetorun --channel=<hex> to execute. No legitimate software combines those exact two flags. This is the single highest-fidelity Amber Albatross detection in the pack.

// HUNT: Amber Albatross - T1622 - --safetorun + --channel= argument pair
// HYPOTHESIS: Required runtime args for the Pyarmor PyInstaller payload.

10. Honey-Token Recipe — Chrome Enrollment + Browser-Profile Canaries

Amber Albatross's most distinctive reconnaissance action is reading HKLM\SOFTWARE\Policies\Google\Chrome\CloudManagementEnrollmentToken. We can turn this into a zero-FP detection trap by planting a synthetic enrollment token and a synthetic browser profile, then alerting on any read.

Step 1 — Plant a synthetic Chrome enrollment token on hosts that don't legitimately use one

On every Windows host that is NOT actually enrolled in Chrome Browser Cloud Management:

# Run as Administrator. Idempotent.
$path = 'HKLM:\SOFTWARE\Policies\Google\Chrome'
if (-not (Test-Path $path)) { New-Item -Path $path -Force | Out-Null }
# A clearly-fake but realistic-looking token (UUID + suffix)
$canary = 'a3f4e5d6-9c2b-4f1a-CANARY-DO-NOT-USE-1' + (Get-Random)
Set-ItemProperty -Path $path -Name 'CloudManagementEnrollmentToken' -Value $canary -Type String
Write-Host "Canary CME token planted at $path. Value: $canary"

# Mark this value in your asset inventory as a canary.
# If Chrome reads it (because the user actually enrolled the host), DELETE the canary immediately.

Step 2 — Audit-on-read SACL via PowerShell (works alongside Falcon registry-read telemetry)

# Enable registry auditing (one-time per host, or via Local Security Policy / GPO)
auditpol /set /subcategory:"Registry" /success:enable /failure:enable

# Apply Audit SACL to the canary key (PowerShell)
$path = 'HKLM:\SOFTWARE\Policies\Google\Chrome'
$key  = [Microsoft.Win32.Registry]::LocalMachine.OpenSubKey(
          'SOFTWARE\Policies\Google\Chrome',
          [Microsoft.Win32.RegistryKeyPermissionCheck]::ReadWriteSubTree,
          [System.Security.AccessControl.RegistryRights]::ChangePermissions)
$audit = New-Object System.Security.AccessControl.RegistryAuditRule(
           "Everyone",
           [System.Security.AccessControl.RegistryRights]::QueryValues,
           [System.Security.AccessControl.AuditFlags]::Success)
$acl = $key.GetAccessControl([System.Security.AccessControl.AccessControlSections]::Audit)
$acl.AddAuditRule($audit)
$key.SetAccessControl($acl)
$key.Close()
Write-Host "Registry audit SACL applied. Windows Event 4663 will fire on every read."

After this, every read of CloudManagementEnrollmentToken produces a Windows Event ID 4663 (Object Access). Forward to your SIEM and alert.

14. Hardening Recommendations

Single highest-leverage control: if you do nothing else, deploy a WDAC or AppLocker policy that allows only signed binaries from publishers on your enterprise allowlist, and explicitly denies the 11 known Amber Albatross cert thumbprints. The PUP-and-fake-PDF-utility delivery model that drives Amber Albatross's 11.1% customer-affected rate dies the moment Windows refuses to execute unknown publishers.

Immediate — deploy this week (Tier 1)

  1. Import the 11 Amber Albatross cert thumbprints into the GPO Untrusted Certificates store at Computer Configuration > Windows Settings > Security Settings > Public Key Policies > Untrusted Certificates. Closes the door on the published cert pool. MITRE M1038 (Execution Prevention) + M1045 (Code Signing); CIS Microsoft Windows 11 Benchmark §18 (Public Key Policies); MS Security Baseline — Untrusted Certificates list maintenance.
  2. DNS-block the 12 Amber Albatross C2 domains (10 current + 2 historical) at your DNS firewall (e.g., Cisco Umbrella, Cloudflare Gateway, Microsoft DNS Block List, Pi-hole, Quad9 enterprise). MITRE M1031 (Network Intrusion Prevention) + M1037 (Filter Network Traffic); CIS Microsoft Windows 11 §9.5 (Domain Profile).
  3. Deploy the §10 honey-token (synthetic CloudManagementEnrollmentToken) on every workstation that isn't actually enrolled in Chrome Browser Cloud Management. Add §10 Step-2 SACL. MITRE Engage EAC0007 (Lures); D3FEND D3-DA (Decoy Artifact).
  4. Promote IOA-1, IOA-3, IOA-5 to Prevent on day 1 (zero-FP profiles per §13).
  5. Enable Microsoft Defender ASR rule D4F940AB-401B-4EFC-AADC-AD5F3C50688A "Block all Office applications from creating child processes" (Amber Albatross PDFast lure occasionally pivots through Office-handled file types). Plus rule 5BEB7EFE-FD9A-4556-801D-275E5FFC04CC "Block execution of potentially obfuscated scripts." MITRE M1050 (Exploit Protection); MS Security Baseline — ASR rules.
  6. Force Microsoft Edge / Chrome to enterprise-management mode: ensure HKLM\SOFTWARE\Policies\Google\Chrome\CloudManagementEnrollmentToken contains a real token on managed hosts; users on unmanaged installs become a recon-target signal worth its own ticket. MITRE M1054 (Software Configuration).

Short-term — 1 to 4 weeks (Tier 2)

  1. WDAC or AppLocker policy in Audit mode → Enforce mode on workstation pilot ring. Block unsigned executables from %LOCALAPPDATA%, %APPDATA%, %TEMP%, %PROGRAMDATA%. Strongly inhibits the PUP→stealer drop pattern. MITRE M1038; CIS Microsoft Windows 11 §18.10.42 (AppLocker).
  2. Enable Chrome AppBound encryption (Chrome 127+) to protect saved passwords and cookies from in-user-context stealers. MITRE M1041 (Encrypt Sensitive Information).
  3. Enable Microsoft Defender SmartScreen for Edge AND for OS at the Block (not Warn) level via GPO. Forces installers without strong reputation to be blocked outright, not just warned about. MITRE M1021 (Restrict Web-Based Content); CIS Microsoft Windows 11 §18.9.85.
  4. PowerShell Constrained Language Mode for standard users via WDAC. Defeats the Base64-encoded download cradle outright. MITRE M1038; MS Security Baseline.
  5. Block standard-user installation of third-party software — remove local-admin rights from end-user accounts (LAPS for break-glass). MITRE M1018 (User Account Management) + M1026 (Privileged Account Management); CIS Microsoft Windows 11 §2.3.1.
  6. Promote IOA-2, IOA-4 to Prevent after their 7 / 14 day measurement windows.
  7. User-awareness training on PUP / fake-PDF-utility lures. Most Amber Albatross infections start with a user clicking "Free PDF Compressor" in a search ad. MITRE M1017 (User Training).

Long-term / structural — 1 to 3 months (Tier 3)

  1. Browser-Bound Session Credentials (DBSC) / Token Binding rollout — kills the value of stolen browser cookies to a remote actor. MITRE M1041 + M1056 equivalent.
  2. Centralized browser management via Chrome Browser Cloud Management or Edge for Business — closes the recon signal (CloudManagementEnrollmentToken will be populated and read access is legitimate) AND lets you enforce credential-protection policy fleet-wide. MITRE M1054.
  3. WDAC publisher allowlist for code-signing — explicit allow of only enterprise-approved publishers. Effectively negates GlobalSign EV abuse en bloc, not just per-thumbprint. MITRE M1038 + M1045.
  4. Microsoft Defender for Endpoint + Attack Surface Reduction full ruleset in Block mode. MITRE M1050.
  5. SOC playbook on stealer-credential-disclosure: when a host is confirmed Amber Albatross, IR must rotate Azure / M365 / SSO sessions for the affected user within 1 hour. Stolen cookies expire at the session-token TTL — beat it. MITRE M1032 (Multi-factor Authentication).
  6. Periodic re-pull of TDR cert + domain list: monthly automation reads Red Canary's TDR page, diffs against your block list, opens a ticket if new thumbprints / domains have appeared. No source of truth other than the TDR page itself for the current list.
  7. Asset-criticality scoring applied to stealer hits: a Tier 0 admin workstation hit by Amber Albatross is an immediate-isolate + re-image + creds-rotate event. A Tier 3 dev VM is a same-day re-image. Defines the playbook tempo. MITRE M1018.

15. Containment Runbook (T+0 → T+15 min)

When Q1 / Q3 / Q7 / Q-CANARY fires — or any other strong Amber Albatross signal — follow this sequence. Order matters; the stolen-cookies clock is ticking.

T+0 to T+2 min — Confirm and preserve

  1. Confirm the alert: open detection in Falcon; verify the source process is non-system; verify the host is online.
  2. Page the on-call SOC analyst AND the affected user's Identity team contact. Stealer = stolen browser session cookies = active credential-misuse risk in M365 / Azure / SSO.
  3. Run §9 RTR triage script. Captures Pyarmor PyInstaller process tree, _MEI extraction folder, signing context for recent dropped binaries, DNS cache hits, persistence entries — all of which are volatile.

T+2 to T+8 min — Contain the stolen session

  1. Network-contain the host via Falcon ("Network contain"). Stops further data exfil + new C2 callbacks. Do not yet shut down — keep volatile state.
  2. Rotate the user's M365 / Azure refresh tokens IMMEDIATELY: Revoke-AzureADUserAllRefreshToken -ObjectId <user> (or Entra ID admin centre → user → Sign-ins → Revoke sessions). Force MFA re-prompt on next sign-in.
  3. Rotate the user's SSO tokens at every connected IdP: Okta, Auth0, Duo, Ping, etc. Same logic — stealer's value compounds with every minute the cookie remains valid.
  4. Snapshot the VM (if virtualized) for forensic analysis. Tag with incident ID.

T+8 to T+15 min — Stop the bleed; pivot to siblings

  1. Identify the lure: which PUP / fake PDF utility did the user run? Check browser history; check most recent downloads; check installer parent process from the RTR bundle. Document the URL.
  2. Pivot to siblings: search the rest of the fleet for (a) the same C2 domain in DNS logs, (b) the same cert thumbprint in file-write events, (c) the same lure URL in browser history. Use Q3 + Q11 with the new pivot.
  3. Block the lure URL and parent domain at the proxy + DNS firewall; submit to your enterprise URL-reputation feed.
  4. Force password change for any account whose credentials might have been in any browser on this host. Especially: AD password, M365 password, any SaaS-app password the user routinely signs into.
  5. Document in the incident ticket: alert IDs, lure URL, RTR bundle, snapshot ID, tokens rotated, sibling hosts checked.

T+15 min and beyond — Eradicate & recover

  1. Re-image preferred over remediation: the host had unsigned-trust violations and ran a stealer. Rebuild from gold image. Restore user data from the snapshot AFTER scanning.
  2. Hunt for persistence on the original host (now snapshotted): scheduled tasks, Run keys, WMI permanent subscriptions, services. All captured in the RTR bundle.
  3. Add the new IOCs (lure URL, any new cert thumbprint, any new file hash) to the §12 IOC table and re-import to Falcon.
  4. Brief the user: phishing-awareness moment, especially on PUP and fake-PDF lures. Most users don't realize "free PDF compressor" is a stealer vector.
  5. Post-incident review at 24 / 72 / 168 hours: what fired? What missed? Update Q1–Q15 + IOA tuning. Update training material with the actual lure URL.
Anti-pattern to avoid: re-image first, rotate tokens second. The stealer's gold isn't on the host — it's the browser cookie that's already in the actor's pocket. Token rotation is the FIRST containment step, not the last.
