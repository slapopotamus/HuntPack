Executive Summary

FortiBleed is an active campaign uncovered by SOCRadar (published 16 Jun 2026) after researchers located the operators' exposed staging server. The actor systematically scans the internet for Fortinet FortiGate firewalls and SSL-VPN gateways on common deployment ports (443, 4443, 8443, 10443), credential-stuffs each one with a curated list of passwords previously leaked from Fortinet devices, and records every successful login into a verified, working-credential database — 30,791 access points across 21,108 IPs, 8,316 domains, and 194 countries.

The defining behavior is the feedback loop: once a device is compromised, the actor turns it into a passive listening post, sniffing credentials out of traffic transiting the firewall and feeding those credentials back into the scanner. Post-exploitation observed on the staging server included extraction of SSL-VPN and admin credentials, network topology, firewall policies and backup configs, plus inward pivoting that used Java malware launched via DLL side-loading to exfiltrate NTDS.dit and the SYSTEM registry hive. Volume on the server: ~1.16 billion credential attempts against 320,000+ FortiGate targets (plus 2.1 billion brute-force attempts against 160,000+ MSSQL servers). Targeting skews heavily toward NATO-member countries and critical sectors (telecom, government, energy, finance).

The hard truth for detection: the highest-fidelity signal lives on the appliance, which CrowdStrike Falcon EDR does not run on. This pack therefore splits into two halves — (a) FortiGate syslog ingested into Falcon LogScale / Next-Gen SIEM for the auth-abuse and on-device activity, and (b) native Falcon EDR telemetry for the inward pivot (credential dumping, lateral movement, exfil) on Windows hosts the actor reaches with harvested credentials.

Defender priority: If a FortiGate appears in the SOCRadar dataset, treat the perimeter as already breached — rotate every FortiGate admin/VPN/service credential, pull mgmt off the public internet, and enforce MFA + client-certificate auth today. Detection is necessary but secondary to credential rotation, because the actor already holds verified working logins.

CrowdStrike LogScale CQL Hunt Queries

Queries Q3–Q5 and Q7 parse FortiGate syslog ingested into LogScale; their field names (action, status, user, srcip, logid, logdesc, cfgpath) come from the FortiOS key/value log format and depend on your Fortinet parser. Validate them against your tenant's parsed fields before deploying. Q1–Q2, Q6, Q8–Q9 use native Falcon EDR events.
Q1 · Beacon to FortiBleed staging IP
CONF HIGHFP LOWCOST LOW

Looks for: any host connecting outbound to the known operator staging server. Accomplishes: a fast, high-confidence IOC sweep for inward-pivot exfil or C2.

// HUNT: Beacon to FortiBleed staging IP
// MITRE: T1071, T1567
// CONF: high   FP: low   COST: low
// REQUIRES: NetworkConnectIP4 telemetry
// FALSE POSITIVES: none expected (specific attacker IP); IP may be rotated/reused over time
#event_simpleName=NetworkConnectIP4
| RemoteAddressIP4="85.11.187.8"
| groupBy([ComputerName, aid, RemoteAddressIP4, RemotePort], function=[count(as=conns), min(@timestamp, as=first), max(@timestamp, as=last)])
| sort(conns, order=desc)
Q2 · NTDS.dit / SYSTEM hive extraction
CONF HIGHFP MEDCOST MED

Looks for: the tools the actor used to extract the domain credential store. Accomplishes: catches the inward-pivot credential-dumping step (the most damaging post-VPN action).

Common FP: ntdsutil.exe / vssadmin.exe — legitimate AD backups and DR tests. Exclude scheduled backup service accounts and known backup hosts.
// HUNT: NTDS.dit / SYSTEM hive extraction
// MITRE: T1003.003
// CONF: high   FP: medium   COST: medium
// REQUIRES: ProcessRollup2 with command line
// FALSE POSITIVES: legitimate AD backup/DR (ntdsutil, vssadmin) by backup accounts
// TUNING: exclude UserName in known backup service accounts and dedicated backup hosts
#event_simpleName=ProcessRollup2
| (ImageFileName=/\\(ntdsutil|vssadmin|esentutl|wbadmin)\.exe$/i
   or CommandLine=/ntds\.dit/i
   or CommandLine=/(create\s+full|ifm|ac\s+i\s+ntds)/i
   or CommandLine=/reg\s+save\s+.*\\(SYSTEM|SAM|SECURITY)/i)
| table([@timestamp, ComputerName, UserName, ImageFileName, CommandLine, ParentBaseFileName])
| sort(@timestamp, order=desc)
Q3 · FortiGate login success after failure burst
CONF HIGHFP MEDCOST MED

Looks for: a source IP that fails many FortiGate logins then succeeds. Accomplishes: the credential-stuffing → verified-login signature at the heart of FortiBleed.

Common FP: fat-fingered users / expired passwords from corporate egress IPs. Exclude internal/known-good source ranges and require an external srcip.
// HUNT: FortiGate login success after failure burst (credential stuffing)
// MITRE: T1110.004, T1078, T1133
// CONF: high   FP: medium   COST: medium
// REQUIRES: FortiGate VPN/admin auth syslog in LogScale (validate field names vs your parser)
// FALSE POSITIVES: legitimate users retrying after password reset/expiry
// TUNING: exclude trusted corporate egress IPs; require srcip outside RFC1918/known ranges
#repo=fortinet
| @rawstring=/(ssl-login-fail|ssl-login|login\s+failed|administrator|logdesc)/i
| kvParse()
| in(action, values=["ssl-login-fail","ssl-login","login","logout"], ignoreCase=true)
| case {
    action=/fail/i | outcome:="fail";
    status=/fail/i | outcome:="fail";
    * | outcome:="success";
  }
| groupBy([srcip], function=[
    count(as=total),
    count(outcome, distinct=false),
    collect([user, action], limit=25),
    sum(if(outcome=="fail",1,0), as=fails),
    sum(if(outcome=="success",1,0), as=successes)
  ])
| fails>=10 successes>=1
| sort(fails, order=desc)
Q4 · External login to built-in / admin accounts
CONF MEDFP MEDCOST LOW

Looks for: successful FortiGate logins to generic admin / built-in Fortinet system accounts from an external source. Accomplishes: the most-hit credential class in the SOCRadar dataset.

Common FP: legitimate admin login from a remote office. Exclude known admin egress IPs; investigate any external login to default-named accounts.
// HUNT: External login to built-in / admin FortiGate accounts
// MITRE: T1078, T1133
// CONF: medium   FP: medium   COST: low
// REQUIRES: FortiGate auth syslog in LogScale (validate field names vs your parser)
// FALSE POSITIVES: legitimate remote admin sessions
// TUNING: exclude approved admin source IPs; alert only on external srcip
#repo=fortinet
| kvParse()
| in(action, values=["ssl-login","login"], ignoreCase=true)
| (status=/success/i or action=/login$/i)
| user=/^(admin|fgts|fort|fortinet|tech|telm|supp)/i
| !cidr(srcip, subnet="10.0.0.0/8")
| !cidr(srcip, subnet="172.16.0.0/12")
| !cidr(srcip, subnet="192.168.0.0/16")
| groupBy([user, srcip], function=[count(as=logins), collect([devname, dstport], limit=10), max(@timestamp, as=last)])
| sort(logins, order=desc)
Q5 · SSL-VPN auth-failure bursts (stuffing sweep)
CONF MEDFP HIGHCOST MED

Looks for: source IPs generating high volumes of SSL-VPN auth failures across many usernames. Accomplishes: surfaces the scanning/stuffing front-end before any success.

Common FP: internet background noise hits any exposed VPN constantly. Tune fails threshold and distinctUsers up for your baseline; pair with Q3 for the success signal.
// HUNT: SSL-VPN auth-failure bursts (credential-stuffing sweep)
// MITRE: T1110.004, T1595
// CONF: medium   FP: high   COST: medium
// REQUIRES: FortiGate SSL-VPN auth syslog in LogScale (validate field names vs your parser)
// FALSE POSITIVES: constant internet scanning noise against exposed VPN portals
// TUNING: raise fails/distinctUsers thresholds to your baseline; focus on multi-username sources
#repo=fortinet
| @rawstring=/ssl-login-fail|login\s+failed/i
| kvParse()
| action=/fail/i or status=/fail/i
| groupBy([srcip], function=[count(as=fails), count(user, distinct=true, as=distinctUsers), collect(user, limit=30)])
| fails>=50 distinctUsers>=5
| sort(fails, order=desc)
Q6 · Java DLL side-loading from unusual path
CONF MEDFP MEDCOST MED

Looks for: java/javaw executing from or alongside DLLs in user-writable / temp paths. Accomplishes: the DLL side-loading vector used to launch the exfil malware.

Common FP: legitimate Java apps install under AppData. Exclude approved Java vendors/paths and signed images.
// HUNT: Java executing from user-writable / temp path (side-load vector)
// MITRE: T1574.002
// CONF: medium   FP: medium   COST: medium
// REQUIRES: ProcessRollup2 with command line and image path
// FALSE POSITIVES: legitimate per-user Java app installs
// TUNING: exclude approved Java install paths and signed publisher hashes
#event_simpleName=ProcessRollup2
| ImageFileName=/\\(java|javaw)\.exe$/i
| ImageFileName=/\\(Users|Temp|ProgramData|AppData|Public|Downloads)\\/i
| table([@timestamp, ComputerName, UserName, ImageFileName, CommandLine, ParentBaseFileName, SHA256HashData])
| sort(@timestamp, order=desc)
Q7 · FortiGate config backup / change post-auth
CONF MEDFP MEDCOST LOW

Looks for: config-backup, config-edit, or sniffer-related events on the FortiGate. Accomplishes: the on-device collection / config-exfil step (partial coverage — sniffer capture itself is not logged).

Common FP: scheduled config backups and authorized admin changes. Exclude change-window timestamps and approved admin accounts.
// HUNT: FortiGate config backup / change / CLI activity post-auth
// MITRE: T1602, T1040
// CONF: medium   FP: medium   COST: low
// REQUIRES: FortiGate event/config syslog in LogScale (validate field names vs your parser)
// FALSE POSITIVES: scheduled backups, authorized admin changes
// TUNING: exclude approved admin accounts and scheduled-backup windows
#repo=fortinet
| @rawstring=/(Configuration backed up|cfgpath|backup|cli\s+command|diagnose\s+sniffer)/i
| kvParse()
| (logdesc=/backup|configuration|cli command/i or cfgpath=* or msg=/sniffer/i)
| groupBy([user, srcip, logdesc], function=[count(as=events), collect([cfgpath, msg], limit=15), max(@timestamp, as=last)])
| sort(events, order=desc)
Q8 · Internal network-logon failure bursts
CONF MEDFP MEDCOST MED

Looks for: Windows hosts seeing bursts of network (type 3) logon failures — the inward MSSQL/service brute-force. Accomplishes: catches lateral credential reuse after the VPN pivot.

Common FP: stale service credentials and misconfigured apps generate steady failures. Baseline per host; alert on spikes above normal.
// HUNT: Internal network-logon failure bursts (lateral brute force)
// MITRE: T1110, T1021
// CONF: medium   FP: medium   COST: medium
// REQUIRES: UserLogonFailed2 telemetry
// FALSE POSITIVES: stale service accounts, misconfigured apps
// TUNING: baseline per ComputerName; exclude known noisy service accounts
#event_simpleName=UserLogonFailed2
| LogonType=3
| groupBy([ComputerName, UserName], function=[count(as=fails), count(RemoteAddressIP4, distinct=true, as=srcCount), collect(RemoteAddressIP4, limit=20)])
| fails>=20
| sort(fails, order=desc)
Q9 · Rare interactive/remote logon (pivot)
CONF LOWFP HIGHCOST MED

Looks for: remote-interactive (type 10) logons to hosts a user rarely touches. Accomplishes: low-fidelity backstop for VPN-sourced lateral movement; best run as an investigate hunt, not an alert.

Common FP: IT admins and jump-host usage produce many type-10 logons. Use as an enrichment/triage list, correlate with Q1/Q8.
// HUNT: Remote-interactive logon to rarely-used host (pivot backstop)
// MITRE: T1078, T1021.001
// CONF: low   FP: high   COST: medium
// REQUIRES: UserLogon telemetry
// FALSE POSITIVES: admins, jump hosts, RDP-heavy roles
// TUNING: exclude admin/jump-host accounts; run as investigate hunt not alert
#event_simpleName=UserLogon
| LogonType=10
| groupBy([UserName, ComputerName], function=[count(as=logins), collect(RemoteAddressIP4, limit=10), min(@timestamp, as=first)])
| logins<=2
| sort(first, order=desc)

Hardening — Tiered & Deployable

Immediate — deploy this week
  • Rotate every FortiGate admin, VPN, and service credential (LDAP/RADIUS/SSO bindings included). The actor holds verified working logins; rotation is the only thing that invalidates them. MITRE M1027
  • Pull the management interface off the public internet — restrict admin access via trusted-host / local-in-policy to a management VLAN or jump host. M1035 · CIS FortiGate Benchmark (admin access)
  • Enforce MFA on all VPN and admin accounts (FortiToken / FIDO2). Makes a stolen password insufficient. M1032
  • Set login-attempt limits + block time on SSL-VPN to blunt stuffing. M1036
  • Block egress to 85.11.187.8 at the perimeter and add to IOC management. M1037
Near term — 1–4 weeks (pilot first)
  • Require client-certificate auth on SSL-VPN so the connection never reaches a password prompt without a valid cert. M1032
  • Geo / IP allowlisting via local-in-policy on the VPN portal — drop the bulk of automated traffic before the login page. M1037
  • Patch FortiOS and FortiClient EMS to the latest fixed builds — closes CVE-2024-21762, CVE-2026-24858, CVE-2026-35616 used as alternate access paths. M1051
  • Rename or disable default/built-in and unused accounts (the admin/fort*/fgts* classes the actor targets first). M1018 / M1027
  • Ingest FortiGate syslog into LogScale so Q3–Q7 actually have data. M1047
Strategic — 1–3 months (architectural)
  • Migrate SSL-VPN to ZTNA with continuous device + user authentication, retiring the always-on perimeter gateway. M1035
  • Network segmentation / micro-segmentation so a compromised edge device cannot reach the domain core. M1030
  • Move toward phishing-resistant passwordless auth (FIDO2 / certificate-based) org-wide to neutralize credential-stuffing as a class. M1032
  • Continuous NDR / egress monitoring on traffic transiting edge appliances for sniffer-style behavior. M1047 ⚠ best-practice, no formal benchmark for on-device sniffer detection

Playbook A — FortiOS SSL-VPN & admin hardening (FortiGate CLI)

    edit "wan1"

    next

    set gui-allow-unauth disable
end

Containment Runbook

Phase 1 — Isolate
  • Block egress to 85.11.187.8 at the perimeter; add to IOC management. (Owner: NetSec · Evidence: firewall/IOC change ticket)
  • Disable the SSL-VPN portal on any FortiGate appearing in the dataset until rebuilt. (Owner: NetSec)
  • Network-contain (Falcon RTR) any endpoint hit by Q1/Q2/Q8. (Owner: SOC)
Phase 2 — Eradicate
  • Treat affected FortiGates as fully compromised: do not just change the password — re-flash firmware to a known-good FortiOS build and rebuild config by hand (do not restore the on-device backup, which may carry attacker changes). (Owner: NetSec · Evidence: forensic image taken first)
  • Rotate all FortiGate admin/VPN credentials, regenerate VPN/server certificates, and rotate every service account bound to the device (LDAP/RADIUS/SSO). (Owner: IAM)
  • On hosts that triggered NTDS/dumping hunts, assume domain credential exposure: rotate krbtgt (twice), reset privileged and service accounts, and review for persistence. (Owner: IR + IAM)
  • Patch FortiClient EMS (CVE-2026-35616) and audit any endpoint software pushed through it. (Owner: Endpoint)
Phase 3 — Recover
  • Re-enable SSL-VPN only with MFA + client-cert and management off the public internet. (Owner: NetSec)
  • Enable FortiGate syslog → LogScale and stand up Q3–Q7 as Scheduled Searches before reopening access. (Owner: Detection Eng)
  • Monitor for 30 days: re-run Q1–Q9 daily; confirm no recurrence of staging-IP egress or stuffing bursts. (Owner: SOC · Evidence: clean daily hunt results)
