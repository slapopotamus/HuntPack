Executive Summary

Kali365 is an emerging Phishing-as-a-Service (PhaaS) platform that the FBI/IC3 warned about in PSA260521. First observed in April 2026 and distributed mainly via Telegram (reportedly ~$250/month or ~$2,000/year), it is a turnkey kit that lets even non-technical actors run sophisticated campaigns — AI-generated phishing lures, automated templates, real-time victim-tracking dashboards, and OAuth token capture. Researchers documented hundreds of attacks in April alone across North America and Europe.

The core technique is device-code phishing — abuse of Microsoft's legitimate OAuth 2.0 Device Authorization Grant. The attacker initiates a device-code flow on their own machine, obtains a user_code, and emails the victim that code plus a link to Microsoft's real Device Authorization page (microsoft.com/devicelogin). The victim — believing they are completing a normal sign-in — authenticates and enters the attacker's code, thereby authorizing the attacker's device. Microsoft then issues OAuth access and refresh tokens to the attacker. Because the victim performed a genuine, MFA-satisfied login on Microsoft's own domain, MFA is satisfied and bypassed, and no password is ever stolen. The refresh token gives durable access until it is revoked.

⚠ Defender priority & the fix: The single most effective control is a Conditional Access policy that blocks the device-code authentication flow for all users (with scoped exceptions + emergency-access exclusions) — the FBI's #1 recommendation. Then hunt N1/Q1 (device-code-flow sign-ins, especially where that flow isn't expected) and Q2 (token use from a different IP/ASN than the grant). Detection here lives in Entra ID / M365 sign-in & audit logs, not endpoint telemetry.

CrowdStrike LogScale CQL Hunt Queries

⚠ These queries target Microsoft 365 / Entra ID sign-in & audit logs ingested into Falcon Next-Gen SIEM (e.g. via the Microsoft 365 / Entra data connector) — not endpoint sensor events. Field names (vendor.* / parser fields) depend on your connector's parser; the patterns below use common Entra field names — validate against your repo schema before deploying. If you don't ingest identity logs, use the native hunts in §6.
Q1 · Device-code authentication flow sign-ins
CONF HIGHFP LOW-MEDCOST LOW

Looks for: Entra sign-ins using the device-code flow. Accomplishes: the flagship Kali365 signal. FP: low-med — legit device-code use exists (CLI tools, IoT, smart-TV apps); baseline which users/apps legitimately use it and alert on the rest.

// HUNT: Entra ID device-code authentication flow sign-ins (Kali365)
// MITRE: T1528, T1621 | CONF: high  FP: low-med  COST: low
// REQUIRES: M365/Entra sign-in logs in NG-SIEM | validate field names vs your parser
// FP NOTES: CLI/IoT/smart-TV apps use device code legitimately; baseline + allow-list
#repo="m365"
| vendor.category=/SignInLogs/i
| vendor.properties.authenticationProtocol=/deviceCode/i
| table([@timestamp, vendor.properties.userPrincipalName, vendor.properties.appDisplayName, vendor.properties.ipAddress, vendor.properties.location.countryOrRegion, vendor.properties.status.errorCode])
Q2 · Token use from a different IP/country than the device-code grant
CONF HIGHFP MEDCOST MED

Looks for: a user whose device-code sign-in and subsequent token-based sign-ins originate from different countries/ASNs. Accomplishes: stolen-token use. FP: medium — VPN/travel; require distinct countries and correlate to Q1.

// HUNT: device-code grant then token use from a different geo (stolen token)
// MITRE: T1078.004 | CONF: high  FP: medium  COST: med
// TUNING: VPN/travel cause FPs; require distinct countries; correlate to Q1
#repo="m365"
| vendor.category=/SignInLogs/i
| vendor.properties.userPrincipalName=~/.+/
| groupBy([vendor.properties.userPrincipalName], function=[
    collect([vendor.properties.location.countryOrRegion], separator=", ", as=countries),
    count(vendor.properties.location.countryOrRegion, distinct=true, as=geo_count)])
| geo_count >= 2
| table([vendor.properties.userPrincipalName, geo_count, countries])
Q3 · New OAuth app consent / service-principal credential add
CONF MEDFP MEDCOST LOW

Looks for: OAuth consent grants or app/SP credential additions in the Entra audit log (post-token persistence). FP: medium — legit admin app onboarding; correlate to a device-code sign-in on the same principal.

// HUNT: post-auth OAuth consent / SP credential add (persistence)
// MITRE: T1098.001 | CONF: medium  FP: medium  COST: low
// TUNING: correlate to a device-code sign-in (Q1) on the same actor/window
#repo="m365"
| vendor.category=/AuditLogs/i
| vendor.properties.activityDisplayName=/(Consent to application|Add app role assignment|Add service principal credentials|Add delegated permission grant)/i
| table([@timestamp, vendor.properties.initiatedBy.user.userPrincipalName, vendor.properties.activityDisplayName, vendor.properties.targetResources.displayName])
Q4 · Inbox forwarding rule / MFA method change post-sign-in
CONF MEDFP MEDCOST LOW

Looks for: new inbox rules (forward/delete) or new MFA methods registered — common post-ATO actions. FP: medium — users do create rules; flag forwarding to external domains and MFA adds right after a device-code sign-in.

// HUNT: post-ATO inbox rule / MFA method change
// MITRE: T1114.003, T1556.006 | CONF: medium  FP: medium  COST: low
// TUNING: flag external-forward rules + MFA adds within hours of a device-code sign-in
#repo="m365"
| vendor.operation=/(New-InboxRule|Set-InboxRule|Add-MailboxPermission|Set-Mailbox|registerSecurityInfo|Update user.+StrongAuthentication)/i
| table([@timestamp, vendor.userId, vendor.operation, vendor.clientIP, vendor.parameters])
Q5 · Anomalous-ASN non-interactive sign-in to M365
CONF MEDFP MEDCOST MED

Looks for: non-interactive (token-based) sign-ins from hosting/VPS ASNs not normal for the user. Accomplishes: attacker token replay from infrastructure. FP: medium — baseline normal ASNs; pair with Q1/Q2.

// HUNT: non-interactive M365 sign-in from a hosting/VPS ASN
// MITRE: T1078.004 | CONF: medium  FP: medium  COST: med
// TUNING: baseline normal ASNs per user; flag hosting providers; pair with Q1/Q2
#repo="m365"
| vendor.category=/(NonInteractiveUserSignInLogs|SignInLogs)/i
| vendor.properties.isInteractive=false
| vendor.properties.autonomousSystemNumber=~/.+/
| groupBy([vendor.properties.userPrincipalName, vendor.properties.autonomousSystemNumber, vendor.properties.ipAddress], function=count())
| table([vendor.properties.userPrincipalName, vendor.properties.autonomousSystemNumber, vendor.properties.ipAddress, _count])
Q6 · Device-code sign-in success summary by app + country
CONF MED-HIGHFP LOWCOST LOW

Looks for: a triage rollup of successful device-code sign-ins grouped by app + country to spot clusters. Accomplishes: a fast hunt overview to find which apps/geos are seeing device-code abuse. FP: low — it's a summary; investigate outliers.

// HUNT: device-code sign-in rollup by app + country (triage overview)
// MITRE: T1528 | CONF: med-high  FP: low  COST: low
// TUNING: investigate apps/countries you don't expect to use device code
#repo="m365"
| vendor.category=/SignInLogs/i
| vendor.properties.authenticationProtocol=/deviceCode/i
| vendor.properties.status.errorCode=0
| groupBy([vendor.properties.appDisplayName, vendor.properties.location.countryOrRegion], function=count(as=signins))
| sort(signins, order=desc)

Hardening — Tiered & Deployable

The decisive control is preventive: block the device-code flow in Conditional Access. Detection backs up any scoped exceptions.

Immediate — Block the flow (the fix)
  • Conditional Access policy to BLOCK the device-code authentication flow for all users — M1037 / M1018. The FBI's #1 recommendation. Audit device-code usage first (N3) to scope exceptions, and exclude emergency-access (break-glass) accounts to avoid lockout.
  • Block authentication transfer (PC → mobile) — M1037. Closes the related transfer abuse the FBI calls out.
  • User awareness: never enter a device code someone sent you — M1017. The lure dies if the victim doesn't enter the code.
  • Promote Q1 (device-code sign-ins) + Q2 (cross-geo token use) to alerts — M1047.
Near term — Constrain tokens (pilot first)
  • Phishing-resistant MFA (FIDO2 / passkeys); token protection + Continuous Access Evaluation — M1032. Binds tokens to device/session so a stolen refresh token is less useful.
  • Sign-in & user risk policies that revoke refresh tokens on risk; shorten token lifetimes — M1036.
  • App governance / OAuth consent policies (admin consent for risky scopes) — M1018. Blunts post-token persistence (Q3).
Strategic — Reduce identity attack surface
  • Enforce phishing-resistant MFA org-wide; retire OTP/push where feasible — M1032.
  • Entra ID Protection + UEBA on sign-in anomalies; automated session revocation — M1047.
  • Mailbox auditing on by default; alert on external-forward inbox rules — M1047.

Deployable — Conditional Access: block device code flow (Microsoft Graph)

POST https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies
Content-Type: application/json

{
  "displayName": "BLOCK - Device Code Flow (Kali365 mitigation)",
  "state": "enabledForReportingButNotEnforced",   // pilot first, then set to "enabled"
  "conditions": {
    "users": { "includeUsers": ["All"], "excludeUsers": [""] },
    "applications": { "includeApplications": ["All"] },
    "authenticationFlows": { "transferMethods": "deviceCodeFlow,authenticationTransfer" }
  },
  "grantControls": { "operator": "OR", "builtInControls": ["block"] }
}
// Validate in report-only mode against the §6/N3 device-code audit before enforcing.
