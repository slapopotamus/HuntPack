1. Executive Summary

NetSupport Manager is a legitimate remote administration suite that adversaries repeatedly repurpose as a remote access trojan. Red Canary ranks it as the fourth most prevalent 2025 threat in the Threat Detection Report, affecting 4.5 percent of customers, and notes that activity increased by 50 percent year over year. Its abuse is hard to treat as a pure malware problem because the same signed software can be used by legitimate IT teams.

The most durable defender strategy is to baseline approved RMM use and hunt for deviations: NetSupport running from ProgramData, AppData, Downloads, or random user directories; renamed NetSupport Client binaries with internal metadata like client32; attacker-controlled client32.ini or client32u.ini files; persistence via HKCU Run keys, UserInitMprLogonScript, Startup links, and scheduled tasks; and delivery chains using JavaScript, PowerShell, curl, msiexec, mshta, finger, forfiles, WMI, archives, and BAT/VBS scripts.

Do not blindly block all NetSupport: Many organizations legitimately use NetSupport Manager or NetSupport School. Exact known-bad domains and hashes are block candidates, but signer/product/path detections should usually begin as detect-only until approved RMM inventory is known.

Observed delivery and abuse chain

       |

       |
       +--> [wscript/cscript executes updateinstaller.js or run.js]

       +--> [PowerShell downloads and extracts NetSupport bundle]
       +--> [msiexec /i http(s)://... /qn or mshta http(s)://...]

       |
       v
[Bundle drops NetSupport files: client32.exe, client32.ini/client32u.ini, NSM.LIC, DLLs]

       v

       +--> [Persistence: HKCU Run, UserInitMprLogonScript, Startup .lnk/.url, scheduled task]

       +--> [Remote operator uploads/downloads files and executes commands]
       |
       v
[Follow-on actions: credential theft, Impacket/WMI/RDP lateral movement, data staging, additional RATs/stealers/ransomware]

7. CrowdStrike LogScale CQL Hunt Queries

CQL review note: Field names can vary by Falcon parser and product tier. Validate with a small time range first. Add your approved NetSupport paths, gateways, and signer allow-list before scheduling broad RMM detections.
Q1NetSupport client32 executing from non-standard directories
CONF: HIGH FP: MED COST: LIGHT
MITRE: T1219, T1036
Requires: Process telemetry
Timeframe: 30-90 days
Promotion: Detect-only after allow-list

Hypothesis: NetSupport normally belongs in Program Files; execution from ProgramData, AppData, Temp, Downloads, or Public is suspicious.

Open in Falcon
#event_simpleName=/^(ProcessRollup2|SyntheticProcessRollup2)$/
| in(field=event_platform, values=["Win"])
| FileName=/^(client32|client64)\.exe$/i
| ImageFileName=/\\(ProgramData|Users\\[^\\]+\\AppData\\(Roaming|Local|LocalLow)|Users\\Public|Windows\\Temp|Temp|Downloads)\\/i
| ImageFileName!=/\\Program Files( \(x86\))?\\NetSupport\\/i
| table([@timestamp, aid, ComputerName, UserName, FileName, ImageFileName, ParentBaseFileName, CommandLine, CertificateSubject, SHA256HashData], limit=200)
Triage: Collect config files in the same directory, signer, hash, parent process, persistence, and network destinations.

11. Hardening

Immediate

  • Inventory all NetSupport and other RMM installations; assign owners and approved gateways.
  • Block or alert on NetSupport from AppData, ProgramData, Public, Temp, and Downloads paths.
  • Block aged campaign domains/IPs where still resolvable and not business-required.
  • Disable public exposure of NetSupport Gateway unless explicitly required and access-controlled.
  • Contain hosts where unauthorized NetSupport has remote-control traffic or persistence.

Near term

  • Implement application control for approved RMM publishers, paths, and deployment tools.
  • Monitor Run keys, UserInitMprLogonScript, Startup folders, and scheduled tasks for RMM payloads.
  • Restrict outbound ports/destinations used by approved NetSupport to known gateways.
  • Harden browser/fake update and ClickFix vectors through DNS filtering, web isolation, and user training.

Strategic

  • Maintain an enterprise RMM registry with owner, vendor, purpose, allowed paths, allowed domains, and emergency contacts.
  • Correlate RMM detections with identity, SMB, RDP, WMI, and data-staging logs.
  • Continuously test coverage with Atomic Red Team tests for PowerShell, Ingress Tool Transfer, Msiexec, Mshta, and Remote Access Software.
  • Adopt zero-trust remote access patterns for IT support tooling and remove unmanaged remote-control tools.

13. Containment Runbook

1. Confirm whether NetSupport is approved on the host.
2. If unauthorized and active, isolate host before killing the process.
3. Preserve NetSupport directory, config files, hashes, signer, process tree, DNS, network, persistence, and browser/proxy precursor logs.
4. Extract gateway/controller values from client32.ini/client32u.ini.
5. Disable HKCU Run, UserInitMprLogonScript, Startup links, and scheduled tasks after evidence capture.
6. Review RDP, SMB, WMI, admin-share, and identity logs from first execution onward.
7. If credential dumping, NTDS access, or operator-driven file staging is present, assume account compromise and rotate credentials.
8. Reimage unauthorized NetSupport hosts when post-exploitation cannot be bounded.
9. Add approved RMM allow-list and deploy detections from Q1-Q18.
